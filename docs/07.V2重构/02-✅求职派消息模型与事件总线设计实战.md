# ✅求职派消息模型与事件总线设计实战

> 消息模型与事件总线是求职派通道层与业务层解耦的核心基础设施。本文完整记录消息模型的设计思路、事件总线的实现机制与使用指南。

---

## 一、为什么需要统一消息模型与事件总线？

### 1.1 多渠道消息的复杂性

求职派需要对接微信、钉钉、飞书等多种 IM 渠道，每种渠道的消息格式差异巨大：

| 渠道 | 消息格式 | 特点 |
|------|---------|------|
| 微信 ClawBot | `WeixinTypes.WeixinMessage` | 轮询获取，包含 `itemList` 数组 |
| 钉钉 | `ChatbotMessage` | WebSocket 推送，支持 AI 卡片 |
| 飞书 | `P2MessageReceiveV1` | 事件驱动，富文本支持 |

如果不做统一抽象，业务层（Agent、Router）需要处理三种不同的消息模型，代码会非常混乱。

### 1.2 设计目标

> **如何让业务层无需关心消息来自哪个渠道，以统一的方式处理所有 IM 消息？**

答案就是：
1. **统一消息模型**：`ChannelReceiveMessage` 和 `ChannelResponseMessage`
2. **事件总线**：`ChannelEventPublisher` 解耦通道层与业务层

---

## 二、消息模型设计：为什么需要统一？

### 2.1 架构概览

```
┌──────────────────────────────────────────────────────┐
│              通道层 (channels/)                         │
│   微信 SDK 消息    钉钉 SDK 消息    飞书 SDK 消息       │
├──────────────────────────────────────────────────────┤
│         消息适配 (adaptToReceive)                      │
│   SDK 消息 → ChannelReceiveMessage（统一接收模型）     │
├──────────────────────────────────────────────────────┤
│              业务层 (agents/router)                     │
│   处理 ChannelReceiveMessage，生成 ChannelResponseMessage │
├──────────────────────────────────────────────────────┤
│         消息回写 (responseToUser)                      │
│   ChannelResponseMessage → 各渠道 SDK 消息格式         │
└──────────────────────────────────────────────────────┘
```

### 2.2 ChannelReceiveMessage（接收消息）：解决什么问题？

#### 为什么需要统一接收消息模型？

**问题场景**：

如果不做统一抽象，业务层需要处理三种完全不同的消息格式：

```java
// 反例：业务层直接处理 SDK 消息
public class MsgRouter {
    public void processMessage(Object sdkMessage) {
        if (sdkMessage instanceof WeixinTypes.WeixinMessage) {
            // 微信消息处理逻辑
            WeixinTypes.WeixinMessage wxMsg = (WeixinTypes.WeixinMessage) sdkMessage;
            String text = MessageBuilder.extractText(wxMsg);
            String userId = wxMsg.getFromUserId();
            // ... 100 行微信特定逻辑
        } else if (sdkMessage instanceof ChatbotMessage) {
            // 钉钉消息处理逻辑
            ChatbotMessage ddMsg = (ChatbotMessage) sdkMessage;
            String text = ddMsg.getText();
            String userId = ddMsg.getSenderStaffId();
            // ... 100 行钉钉特定逻辑
        } else if (sdkMessage instanceof P2MessageReceiveV1) {
            // 飞书消息处理逻辑
            // ... 100 行飞书特定逻辑
        }
    }
}
```

**问题分析**：
1. **代码混乱**：业务逻辑与渠道特定代码混在一起
2. **难以扩展**：每新增一个渠道，都要修改业务层代码
3. **无法复用**：身份采集、意图识别等通用逻辑无法复用
4. **难以测试**：需要构造不同渠道的 SDK 消息对象

#### ChannelReceiveMessage 的设计价值

所有通道接收到的消息统一转换为这个模型：

```java
@Data
@SuperBuilder(toBuilder = true)
@NoArgsConstructor
public class ChannelReceiveMessage {
    private String msgId;                    // 消息唯一 ID
    private String channel;                  // 通道名称
    private String message;                  // 文本消息内容
    private String fromUserId;               // 发送者 ID
    private String jobClawUserId;            // 求职派用户 ID
    
    private List<MediaMsg> medias;           // 媒体消息（图片、视频）
    private List<FileMsg> files;             // 文件消息
    
    private Map<String, Object> passThrough; // 透传参数（渠道个性化信息）
    
    @Builder.Default
    private boolean stream = false;          // 是否流式处理
    
    @Builder.Default
    private boolean groupTalk = false;       // 是否为群聊

    @Data
    @SuperBuilder(toBuilder = true)
    @NoArgsConstructor
    public static class MediaMsg {
        private String downUrl;              // 下载 URL
        private String fileType;             // 文件类型
        private Path filePath;               // 本地文件路径
        private String mimeType;             // MIME 类型
        private byte[] data;                 // 内联数据
    }

    @Data
    @SuperBuilder(toBuilder = true)
    @NoArgsConstructor
    public static class FileMsg {
        private String downUrl;
        private Path filePath;
        private String fileName;
        private String fileType;
        private String mimeType;
    }
}
```

#### 设计要点：为什么这样设计？

**1. 为什么支持多种消息类型？（文本、媒体、文件）**

**问题**：为什么不用一个 `content` 字段存储所有内容？

**答案**：不同消息类型的处理方式完全不同：

| 消息类型 | 处理方式 | 存储位置 |
|---------|---------|----------|
| 文本消息 | 直接传给大模型 | `message` 字段 |
| 图片消息 | 需要下载 → 传给视觉模型 | `medias` 列表 |
| 文件消息 | 需要下载 → 解析内容 | `files` 列表 |

```java
// 业务层处理不同类型消息
if (msg.getMessage() != null) {
    // 文本消息：直接处理
    processText(msg.getMessage());
}
if (msg.getMedias() != null) {
    // 媒体消息：需要调用视觉模型
    for (MediaMsg media : msg.getMedias()) {
        processImage(media.getFilePath());
    }
}
if (msg.getFiles() != null) {
    // 文件消息：需要解析内容
    for (FileMsg file : msg.getFiles()) {
        processFile(file.getFilePath());
    }
}
```

**设计价值**：业务层可以根据消息类型选择不同的处理策略，无需关心消息来自哪个渠道。

**2. 为什么需要 passThrough 透传参数？**

- **文本消息**：`message` 字段存储纯文本内容
- **媒体消息**：`medias` 列表存储图片、视频等多媒体
- **文件消息**：`files` 列表存储文档、表格等文件

**2. 透传参数机制**

`passThrough` 用于传递渠道特有的元数据，原样回传给通道层：

```java
// 微信通道：保存 msgContentToken，用于后续回复
.passThrough(Map.of("msgContentToken", message.getContextToken()))

// 钉钉通道：保存 aiCardId，用于流式更新
.passThrough(Map.of("aiCardId", msg.getAiCardId()))

// 飞书通道：保存 openId 和 messageId
.passThrough(Map.of(
    "openId", openId,
    "messageId", messageId
))
```

**3. 为什么需要 stream 标识？

`stream` 标识是否启用流式响应，主要取决于通道能力：
- 微信 ClawBot：`stream = false`（轮询方式，不支持实时推送）
- 钉钉/飞书：`stream = true`（WebSocket，支持流式更新）

**4. 为什么需要 groupTalk 标识？

`groupTalk` 区分群聊与私聊，影响业务逻辑：
- 群聊：不支持设置用户个人偏好
- 私聊：支持完整的用户画像和偏好管理

#### 消息适配示例

**微信通道适配**：

```java
@Override
public ChannelReceiveMessage adaptToReceive(MsgWrapper<WeixinTypes.WeixinMessage> msgWrapper) {
    WeixinTypes.WeixinMessage message = msgWrapper.getMsg();
    
    // 提取文本
    String messageText = MessageBuilder.extractText(message);
    
    // 提取媒体（图片、文件、视频）
    List<ChannelReceiveMessage.MediaMsg> medias = new ArrayList<>();
    List<ChannelReceiveMessage.FileMsg> files = new ArrayList<>();
    
    for (WeixinTypes.MessageItem item : message.getItemList()) {
        if (item.getImageItem() != null) {
            medias.add(ChannelReceiveMessage.MediaMsg.builder()
                .filePath(item.getImageItem().getLocalPath())
                .mimeType("image/jpeg")
                .build());
        }
        if (item.getFileItem() != null) {
            files.add(ChannelReceiveMessage.FileMsg.builder()
                .fileName(item.getFileItem().getFileName())
                .filePath(item.getFileItem().getLocalPath())
                .build());
        }
    }

    return ChannelReceiveMessage.builder()
            .msgId("WX_" + message.getMessageId())
            .channel(name())
            .fromUserId(message.getFromUserId())
            .jobClawUserId(msgWrapper.getJobClawUserId())
            .passThrough(Map.of("msgContentToken", message.getContextToken()))
            .message(messageText)
            .files(files)
            .medias(medias)
            .stream(false)  // 微信不支持流式
            .build();
}
```

**钉钉通道适配**：

```java
@Override
public ChannelReceiveMessage adaptToReceive(MsgWrapper<ChatbotMessageEx> msgWrapper) {
    ChatbotMessageEx msg = msgWrapper.getMsg();
    
    // 权限校验（OWNER/LOGIN/VIP/PUBLIC）
    String jobClawUserId = resolveUserId(msg, channelConfig);
    if (jobClawUserId == null) return null;
    
    // 解析消息内容
    var msgContent = sdkMap.get(msg.getRobotId()).parseContent(msg);
    
    // 自动下载媒体文件
    if (msgContent.media() != null) {
        var tmpFile = localStorageHelper.autoDownloadFile(
            jobClawUserId, name(), 
            msgContent.media().getDownUrl(), 
            msgContent.media().getFileType()
        );
        msgContent.media().setFilePath(Path.of(tmpFile));
    }

    return ChannelReceiveMessage.builder()
            .msgId(msg.getMsgId())
            .message(msgContent.content())
            .medias(msgContent.media() == null ? null : List.of(msgContent.media()))
            .channel(name())
            .fromUserId(msg.getSenderStaffId())
            .jobClawUserId(jobClawUserId)
            .passThrough(Map.of("aiCardId", msg.getAiCardId()))
            .stream(true)  // ← 钉钉支持流式
            .build();
}
```

### 2.3 ChannelResponseMessage（响应消息）：为什么需要双支持？

#### 为什么需要同时支持同步和流式？

**问题场景**：

用户发送消息后，Agent 处理返回结果：

```
场景 1：简单问答
用户："今天天气怎么样？"
Agent："今天晴天，温度 25°C"  （瞬间返回）

场景 2：复杂任务
用户："帮我分析这份简历，给出优化建议"
Agent：（需要 30 秒处理）
  - 如果不支持流式：用户等待 30 秒，不知道进度
  - 如果支持流式：用户实时看到分析过程
```

**用户体验差异**：

| 响应方式 | 用户感知 | 适用场景 |
|---------|---------|----------|
| 同步响应 | 等待完成后一次性展示 | 简单问答、快速返回 |
| 流式响应 | 实时看到生成过程 | 复杂任务、长文本生成 |

#### ChannelResponseMessage 的设计价值

Agent 处理后的响应消息模型：

```java
@Data
@SuperBuilder(toBuilder = true)
@NoArgsConstructor
@AllArgsConstructor
public class ChannelResponseMessage {
    private String jobClawUserId;                    // 求职派用户 ID
    private String toUserId;                         // 目标用户 ID
    private ResponseMessageType type;                // 消息类型
    private String content;                          // 文本内容（同步）
    private Flux<LlmRspCell> streamContents;         // 流式内容（异步）
    private Map<String, Object> passThrough;         // 透传参数

    public enum ResponseMessageType {
        TEXT("text"),           // 纯文本
        MARKDOWN("markdown"),   // Markdown
        CARD("card"),           // 卡片消息
    }
}
```

#### 设计要点：为什么这样设计？

**1. 为什么用 `content` 和 `streamContents` 两个字段？

- **同步响应**：使用 `content` 字段，适用于一次性返回完整内容
- **流式响应**：使用 `streamContents` 字段，适用于大模型逐步生成

```java
// 同步响应
ChannelResponseMessage.builder()
    .content("您好，我是求职派助手~")
    .type(ResponseMessageType.TEXT)
    .build();

// 流式响应
ChannelResponseMessage.builder()
    .streamContents(llmCaller.stream(conversationInfo, prompt))
    .type(ResponseMessageType.TEXT)
    .build();
```

**2. 为什么需要 type 字段？

`type` 决定通道的渲染方式：
- `TEXT`：纯文本消息（所有通道都支持）
- `MARKDOWN`：Markdown 格式（钉钉、飞书支持）
- `CARD`：卡片消息（钉钉 AI 卡片、飞书交互卡片）

**3. 为什么 passThrough 要原样回传？

`passThrough` 原样回传，便于通道层定位消息接收人：

```java
// MsgRouter 构建响应时，原样携带 passThrough
ChannelResponseMessage responseMessage = ChannelResponseMessage.builder()
    .jobClawUserId(msg.getJobClawUserId())
    .toUserId(msg.getFromUserId())
    .type(ResponseMessageType.TEXT)
    .content(content)
    .passThrough(msg.getPassThrough())  // ← 原样回传
    .build();
```

#### 响应发送示例

**微信通道响应**：

```java
@Override
public boolean responseToUser(ChannelResponseMessage msg) {
    WeixinSdk sdk = accountMap.get(...);
    
    if (msg.getType() == ResponseMessageType.TEXT) {
        if (msg.getStreamContents() != null) {
            // 微信不支持流式，等待流式完成后一次性发送
            String fullContent = msg.getStreamContents()
                .map(LlmRspCell::getContent)
                .collectList()
                .block()
                .stream()
                .collect(Collectors.joining());
            sdk.sendText(msg.getToUserId(), fullContent);
        } else {
            sdk.sendText(msg.getToUserId(), msg.getContent());
        }
    }
    return true;
}
```

**钉钉通道流式响应**：

```java
@Override
public boolean responseToUser(ChannelResponseMessage msg) {
    String aiCardId = (String) msg.getPassThrough().get("aiCardId");
    DingDingSdk sdk = sdkMap.get(...);
    
    if (msg.getStreamContents() != null) {
        // 流式响应：逐步更新 AI 卡片
        msg.getStreamContents().doOnNext(cell -> {
            if (cell.getContent() != null) {
                sdk.updateAiCardContent(aiCardId, cell.getContent());
            }
        }).doOnComplete(() -> {
            aiCardStatus.finishAiCard(...);
        }).subscribe();
    } else {
        // 同步响应
        sdk.sendText(msg.getToUserId(), msg.getContent());
    }
    return true;
}
```

## 三、事件总线设计：为什么需要解耦？

### 3.1 为什么需要事件总线？

**问题**：为什么不直接让 Channel 调用 Agent？

**反例：没有事件总线的架构**

```java
// 通道层直接调用业务层
public class WeChatChannel implements Channel {
    private final MsgRouter msgRouter;  // ← 直接依赖业务层
    
    @Override
    public void reportToAgent(ChannelReceiveMessage msg) {
        msgRouter.onMessageReceived(msg);  // ← 同步调用，阻塞通道
    }
}
```

**问题分析**：

| 问题 | 说明 |
|------|------|
| **强耦合** | 通道层直接依赖业务层，无法独立测试 |
| **阻塞风险** | 同步调用阻塞通道，影响消息接收 |
| **扩展困难** | 新增业务逻辑（如日志、监控）需要修改通道代码 |
| **单点故障** | 业务层异常会导致通道层也异常 |

#### 事件总线的核心价值

事件总线带来以下核心优势：

| 优势 | 说明 |
|------|------|
| **异步处理** | 通道接收消息后立即返回，不阻塞 IM 连接 |
| **解耦** | 通道层无需知道业务层有哪些 Agent |
| **多监听器** | 多个组件可以监听同一事件（日志、监控、审计） |
| **易于测试** | 可以模拟事件进行单元测试 |
| **扩展性强** | 新增业务逻辑无需修改通道代码 |

### 3.2 事件总线架构：为什么这样设计？

```
┌──────────────────────────────────────────────────────┐
│              通道层 (channels/)                         │
│   接收 IM 消息 → adaptToReceive()                     │
│                    ↓                                  │
│         reportToAgent()                               │
├──────────────────────────────────────────────────────┤
│           事件总线 (core/bus/)                          │
│   ChannelEventPublisher.publishMessageReceived()      │
│                    ↓                                  │
│         MessageReceivedEvent (Spring ApplicationEvent) │
├──────────────────────────────────────────────────────┤
│              业务层 (core/router/)                      │
│   MsgRouter.onMessageReceived() @EventListener        │
│                    ↓                                  │
│         身份采集 → 意图识别 → Agent 执行               │
│                    ↓                                  │
│   ChannelEventPublisher.publishMessageResponse()      │
│                    ↓                                  │
│         MessageResponseEvent                          │
├──────────────────────────────────────────────────────┤
│              通道层 (channels/)                         │
│   MsgRouter.onMessageResponse() @EventListener        │
│                    ↓                                  │
│   Channel.responseToUser() → 推送到 IM                │
└──────────────────────────────────────────────────────┘
```

### 3.3 ChannelEventPublisher 接口：为什么需要这些方法？

事件总线解耦了通道层与业务层：

```java
public interface ChannelEventPublisher {

    /**
     * 发布消息接收事件
     * @param channel 通道名称
     * @param message 接收到的消息
     * @param needReply 是否需要回复
     */
    void publishMessageReceived(String channel, ChannelReceiveMessage message, boolean needReply);

    /**
     * 发布消息响应事件
     * @param responseId 响应消息唯一 ID
     * @param relatedMessageId 关联的原始消息 ID
     * @param channel 目标通道名称
     * @param responseMessage 响应消息
     */
    void publishMessageResponse(String responseId, String relatedMessageId, 
                                 String channel, ChannelResponseMessage responseMessage);

    /**
     * 发布主动推送消息（简化版）
     * @param responseId 推送消息 ID
     * @param jobClawUserId 目标用户 ID
     * @param channelName 目标通道名称
     * @param response 推送文本
     */
    boolean publishProactiveMessage(String responseId, String jobClawUserId, 
                                     String channelName, String response);

    /**
     * 发布主动推送消息（流式版）
     */
    boolean publishProactiveMessage(String responseId, String jobClawUserId, 
                                     String channelName, Flux<LlmRspCell> response);

    /**
     * 发布主动推送消息（完整版）
     * @param priority 消息优先级（0=普通，1=高，2=紧急）
     */
    void publishProactiveMessage(String responseId, String channelName, 
                                  ChannelResponseMessage responseMessage, int priority);

    /**
     * 发布用户连接事件
     */
    void publishUserConnected(String channel, String userId, ChannelConfig channelUser, 
                               boolean isNewUser, String sourceIp);

    /**
     * 发布用户断开连接事件
     */
    void publishUserDisconnected(String channel, ChannelConfig channelUser, String reason);
}
```

#### 设计要点：为什么需要这些事件？

**1. 为什么区分 MessageReceivedEvent 和 MessageResponseEvent？**

| 事件 | 触发时机 | 用途 |
|------|---------|------|
| `MessageReceivedEvent` | 通道接收到用户消息 | 触发 Agent 处理流程 |
| `MessageResponseEvent` | Agent 处理完成 | 推送响应到通道 |
| `UserConnectedEvent` | 用户首次绑定通道 | 发送欢迎消息 |
| `UserDisconnectedEvent` | 用户解绑通道 | 清理会话资源 |

**2. 为什么需要三种主动推送方法？

事件总线支持三种主动推送方式：

```java
// 场景 1：定时提醒（同步）
channelEventPublisher.publishProactiveMessage(
    "TASK_" + System.currentTimeMillis(),
    jobClawUserId,
    channelName,
    "您设置的提醒时间到了：面试准备"
);

// 场景 2：岗位推荐（流式）
channelEventPublisher.publishProactiveMessage(
    "RECOMMEND_" + System.currentTimeMillis(),
    jobClawUserId,
    channelName,
    jobRecommendAgent.streamRecommend(conversationInfo)
);

// 场景 3：紧急通知（高优先级）
channelEventPublisher.publishProactiveMessage(
    "ALERT_" + System.currentTimeMillis(),
    channelName,
    ChannelResponseMessage.builder()
        .content("系统维护通知")
        .type(ResponseMessageType.TEXT)
        .build(),
    2  // 紧急优先级
);
```

**3. 为什么需要优先级机制？

主动推送支持优先级，数值越大优先级越高：

| 优先级 | 数值 | 适用场景 |
|--------|------|---------|
| 普通 | 0 | 日常消息、岗位推荐 |
| 高 | 1 | 定时提醒、任务通知 |
| 紧急 | 2 | 系统告警、重要通知 |

### 3.4 事件实现：为什么基于 Spring ApplicationEvent？

#### 为什么选择 Spring ApplicationEvent？

**备选方案对比**：

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **Spring ApplicationEvent** | 简单、内置、与 Spring 生态集成好 | 内存事件，应用重启丢失 | 单体应用、中小规模 |
| **RabbitMQ/Kafka** | 持久化、高可用、分布式 | 复杂、需要额外部署 | 分布式系统、大规模 |
| **Redis Pub/Sub** | 轻量级、支持分布式 | 不持久化、需要额外依赖 | 中小规模、需要分布式 |

**选择 Spring ApplicationEvent 的原因**：
1. 求职派当前是单体应用，不需要分布式事件总线
2. 与 Spring 生态无缝集成，无需额外依赖
3. 实现简单，易于理解和维护
4. 后续如果需要扩展，可以替换为消息队列

基于 Spring `ApplicationEvent` 实现异步事件驱动：

```java
@Component
public class DefaultChannelEventPublisher implements ChannelEventPublisher {
    
    private final ApplicationEventPublisher eventPublisher;

    @Override
    public void publishMessageReceived(String channel, ChannelReceiveMessage message, boolean needReply) {
        MessageReceivedEvent event = new MessageReceivedEvent(this, channel, message, needReply);
        eventPublisher.publishEvent(event);
    }

    @Override
    public void publishMessageResponse(String responseId, String relatedMessageId, 
                                        String channel, ChannelResponseMessage responseMessage) {
        MessageResponseEvent event = new MessageResponseEvent(
            this, responseId, relatedMessageId, channel, responseMessage);
        eventPublisher.publishEvent(event);
    }

    @Override
    public boolean publishProactiveMessage(String responseId, String jobClawUserId, 
                                            String channelName, String response) {
        // 查找用户绑定的通道
        Channel channel = channelRegistry.getChannel(channelName);
        if (channel == null) return false;
        
        // 获取心跳适配器
        Function<Object, ChannelResponseMessage> adapter = 
            channelRegistry.getChannelRspBuilderAdapter(jobClawUserId, channelName);
        if (adapter == null) return false;
        
        // 构建响应消息并发布
        ChannelResponseMessage responseMessage = adapter.apply(response);
        publishMessageResponse(responseId, null, channelName, responseMessage);
        return true;
    }
}
```

### 3.5 事件监听：为什么用 @Async + @EventListener？

#### 为什么需要 @Async？

**问题**：如果不加 @Async 会怎样？

```java
// 反例：同步监听，阻塞通道层
@EventListener  // ← 没有 @Async
public void onMessageReceived(MessageReceivedEvent event) {
    // 身份采集、意图识别、Agent 执行... 可能需要 10 秒
    routeToAgent(agentId, msg, conversationInfo);
}
```

**问题**：
- 通道层会等待事件处理完成才能继续接收下一条消息
- 如果 Agent 执行很慢（如调用大模型），用户发消息后会卡顿
- 对于轮询方式的通道（如微信），可能导致消息丢失

**使用 @Async 的优势**：

```java
@Async  // ← 异步执行，不阻塞通道层
@EventListener
public void onMessageReceived(MessageReceivedEvent event) {
    // 在独立线程中执行，通道层可以继续接收消息
    routeToAgent(agentId, msg, conversationInfo);
}
```

**事件处理流程**：

```
用户发送消息
  ↓
通道层接收 → adaptToReceive()
  ↓
publishMessageReceived() → MessageReceivedEvent
  ↓
Spring 异步线程池分配线程
  ↓
MsgRouter.onMessageReceived() @Async  ← 在独立线程中执行
  ↓
身份采集 → 系统命令 → 意图识别 → Agent 执行
  ↓
publishMessageResponse() → MessageResponseEvent
  ↓
Spring 异步线程池分配线程
  ↓
MsgRouter.onMessageResponse() @Async  ← 在独立线程中执行
  ↓
Channel.responseToUser() → 推送回 IM
```

`MsgRouter` 监听通道事件并路由到具体 Agent：

```java
@Component
public class MsgRouter {

    /**
     * 监听消息接收事件
     */
    @Async
    @EventListener
    public void onMessageReceived(MessageReceivedEvent event) {
        var msg = event.getOriginalMessage();
        String jobClawUserId = msg.getJobClawUserId();
        String fromUserId = msg.getFromUserId();
        String channel = msg.getChannel();
        String userMessage = msg.getMessage();
        
        UserConversationInfo conversationInfo = new UserConversationInfo(
            jobClawUserId, channel, fromUserId, msg.isGroupTalk());

        // Step 1: 身份采集（新用户自动触发）
        if (!msg.isGroupTalk() && identityAgent.triggerToCollectIdentity(conversationInfo, userMessage)) {
            log.info("Message handled by unified initializer for user: {}", jobClawUserId);
            return;
        }

        // Step 2: 系统命令处理
        if (commandDispatcher.isSystemCommand(userMessage)) {
            if (commandDispatcher.executeCommand(msg, conversationInfo, userMessage, 
                                                  str -> sendTextResponse(msg, str))) {
                log.info("System command executed for user: {}", jobClawUserId);
                return;
            }
        }

        // Step 3: 显式切换 Agent（/agent <id>）
        if (userMessage.startsWith("/")) {
            List<BizAgent> bizAgents = agentRegistry.getAllAgents(jobClawUserId);
            for (BizAgent bizAgent : bizAgents) {
                if (userMessage.startsWith("/" + bizAgent.getAgentIntro().getAgentId())) {
                    routeToAgent(bizAgent.getAgentIntro().getAgentId(), msg, conversationInfo);
                    return;
                }
            }
        }

        // Step 4: 判断是否需要意图识别
        String conversationId = conversationInfo.conversationId();
        if (!sessionBinder.needsIntentRecognition(jobClawUserId, conversationId, userMessage)) {
            // 继续使用绑定的 Agent
            String agentId = sessionBinder.getBoundAgentId(jobClawUserId, conversationId).orElse(null);
            routeToAgent(agentId, msg, conversationInfo);
            return;
        }

        // Step 5: 意图识别 + Agent 路由
        IntentClassificationRes classification = intentClassifier.classify(conversationInfo, userMessage);
        AgentRouter.RouterResult routeResult = agentRouter.route(classification, 
            sessionBinder.getBoundAgentId(jobClawUserId, conversationId));

        // Step 6: 绑定会话状态
        if (routeResult.isNewSession()) {
            sessionBinder.bind(jobClawUserId, conversationId, routeResult.agentId());
            sessionBinder.addIntentHistory(jobClawUserId, conversationId, 
                classification.intentType(), classification.confidence());
        }

        // Step 7: 执行 Agent
        routeToAgent(routeResult.agentId(), msg, conversationInfo);
    }

    /**
     * 监听消息响应事件
     */
    @Async
    @EventListener
    public void onMessageResponse(MessageResponseEvent event) {
        var channel = channelRegistry.getChannel(event.getChannel());
        if (channel == null) {
            log.error("找不到对应的通道，请确认这个通道是否正常注册：{}", event.getChannel());
            return;
        }
        channel.responseToUser(event.getResponseMessage());
    }

    /**
     * 监听用户连接事件
     */
    @Async
    @EventListener
    public void onImBotConnected(UserConnectedEvent event) {
        String template = "您已经成功联通求职派啦，现在您可以直接通过对话和求职派进行沟通了~";
        channelEventPublisher.publishProactiveMessage(
            "HI_" + System.currentTimeMillis(),
            event.getUserId(),
            event.getChannel(),
            template);
    }
}
```

#### 设计要点：为什么这样处理事件？

**1. 为什么所有监听器都用 @Async？**

所有事件监听器都使用 `@Async` 注解，确保不阻塞通道层：

```java
@Async  // ← 异步执行，不阻塞 IM 连接
@EventListener
public void onMessageReceived(MessageReceivedEvent event) {
    // ...
}
```

**答案**：确保不阻塞通道层，提升系统响应速度：

```java
@Async  // ← 异步执行，不阻塞 IM 连接
@EventListener
public void onMessageReceived(MessageReceivedEvent event) {
    // ...
}
```

**线程池配置建议**：

```yaml
spring:
  task:
    execution:
      pool:
        core-size: 10      # 核心线程数
        max-size: 50       # 最大线程数
        queue-capacity: 100  # 队列容量
```

**2. 为什么需要降级处理？

消息接收事件的处理流程：
```
用户发送消息
  ↓
通道层接收 → adaptToReceive()
  ↓
publishMessageReceived() → MessageReceivedEvent
  ↓
MsgRouter.onMessageReceived() @Async
  ↓
身份采集 → 系统命令 → 意图识别 → Agent 执行
  ↓
publishMessageResponse() → MessageResponseEvent
  ↓
MsgRouter.onMessageResponse() @Async
  ↓
Channel.responseToUser() → 推送回 IM
```

**3. 降级处理**

如果 Agent 执行失败，回退到 LLM 直接响应：

```java
private void routeToAgent(String agentId, ChannelReceiveMessage msg, 
                           UserConversationInfo conversationInfo) {
    BizAgent agent = agentRegistry.getAgent(agentId)
        .orElseGet(() -> agentRegistry.getDefaultAgent().orElse(null));

    if (agent == null) {
        sendTextResponse(msg, "没有找到可用的Agent哦，请稍后再试。");
        return;
    }

    try {
        String response = agent.process(conversationInfo, msg);
        if (response == null) {
            fallbackToLlm(msg, conversationInfo);  // ← 降级到 LLM
            return;
        }
        sendTextResponse(msg, response);
    } catch (Exception e) {
        log.error("Agent执行失败: {}", agentId, e);
        fallbackToLlm(msg, conversationInfo);  // ← 异常时降级
    }
}
```

## 四、主动推送机制详解：为什么需要心跳？

### 4.1 心跳配置：为什么需要保存？

**问题场景**：

```
用户（通过钉钉）："帮我设置一个明天早上 9 点的面试提醒"
Agent："好的，已为您设置明天 09:00 的面试提醒"

（第二天早上 09:00）
定时任务触发 → 需要推送消息给用户 → 问题来了：
1. 用户通过哪个通道接收消息？（微信？钉钉？飞书？）
2. 如何构建响应消息？（钉钉需要 aiCardId，微信需要 msgContentToken）
```

**如果没有心跳机制会怎样？**

```java
// 反例：后台任务不知道如何推送消息
public void sendReminder(String jobClawUserId, String content) {
    // 问题 1：用户绑定了哪个通道？
    String channelName = ???  // ← 不知道
    
    // 问题 2：如何构建响应消息？
    ChannelResponseMessage response = ???  // ← 不知道需要携带什么透传参数
    
    // 无法推送！
}
```

当用户主动发送消息时，通道会保存心跳信息：

```java
// AbsChannel.processMessage()
var tag = this.saveHeartBeatConfig(msg, force);
if (tag) {
    var func = buildHeartBeatCallback(msg.getJobClawUserId());
    channelRegistry.refreshChannelHeartBeatInfoIgnoreNull(
        msg.getJobClawUserId(), r.getChannel(), func);
}
```

心跳信息包含：
- `jobClawUserId`：求职派用户 ID
- `channelName`：通道名称
- `adapter`：构建响应消息的回调函数

**心跳的本质**：告诉系统"这个用户通过这个通道活跃过，后续可以通过这个通道推送消息"。

### 4.2 主动推送流程：为什么这么复杂？

**问题**：为什么主动推送需要这么多步骤？

**答案**：因为不同通道的推送方式完全不同：

| 通道 | 推送方式 | 需要的信息 |
|------|---------|----------|
| 微信 | ClawBot API 发送 | `msgContentToken` |
| 钉钉 | 更新 AI 卡片 | `aiCardId` |
| 飞书 | 发送消息 API | `openId`, `messageId` |

后台任务（定时提醒、岗位推荐）不知道这些通道特有的信息，所以需要：
1. 通过 `ChannelRegistry` 查找通道
2. 获取通道注册的心跳适配器
3. 适配器由通道层提供，知道如何构建响应
4. 构建响应后发布到事件总线

后台任务主动向用户推送消息的完整流程：

```
后台任务（定时提醒、岗位推荐）
  │
  ▼
channelEventPublisher.publishProactiveMessage()
  │
  ├─ 查找通道：channelRegistry.getChannel(channelName)
  │
  ├─ 获取心跳适配器：channelRegistry.getChannelRspBuilderAdapter(jobClawUserId, channelName)
  │
  ├─ 构建响应：adapter.apply(pushContent) → ChannelResponseMessage
  │
  └─ 发布事件：publishMessageResponse(responseId, null, channelName, responseMessage)
                    │
                    ▼
              MsgRouter.onMessageResponse() @Async
                    │
                    ▼
              Channel.responseToUser() → 推送到 IM
```

### 4.3 心跳适配器示例：为什么需要回调函数？

**问题**：为什么不在保存心跳时直接存储所有信息，而是用回调函数？

**答案**：因为某些信息是动态的，需要在推送时才能确定：

```java
// 反例：保存心跳时存储静态信息
channelRegistry.refreshChannelHeartBeatInfo(
    jobClawUserId, 
    channelName, 
    aiCardId  // ← 问题：aiCardId 可能会过期
);

// 正例：保存心跳时存储回调函数
channelRegistry.refreshChannelHeartBeatInfo(
    jobClawUserId, 
    channelName, 
    content -> {  // ← 推送时才创建新的 aiCardId
        String aiCardId = sdk.initStreamAiCardId(appId, ...);
        return ChannelResponseMessage.builder()
                .passThrough(Map.of("aiCardId", aiCardId))
                .build();
    }
);
```

**回调函数的优势**：
1. 动态生成信息（如创建新的 AI 卡片）
2. 通道层自己决定如何构建响应
3. 后台任务无需关心通道特有的逻辑

**微信通道心跳适配器**：

```java
@Override
public Function<Object, ChannelResponseMessage> buildHeartBeatCallback(String jobClawUserId) {
    return content -> {
        // 查找最新的 msgContentToken
        String msgContentToken = getLatestMsgContentToken(jobClawUserId);
        return ChannelResponseMessage.builder()
                .jobClawUserId(jobClawUserId)
                .toUserId(getToUserId(jobClawUserId))
                .type(ResponseMessageType.TEXT)
                .content(content.toString())
                .passThrough(Map.of("msgContentToken", msgContentToken))
                .build();
    };
}
```

**钉钉通道心跳适配器**：

```java
@Override
public Function<Object, ChannelResponseMessage> buildHeartBeatCallback(String jobClawUserId) {
    return content -> {
        // 创建新的 AI 卡片
        String aiCardId = sdk.initStreamAiCardId(appId, ...);
        return ChannelResponseMessage.builder()
                .jobClawUserId(jobClawUserId)
                .toUserId(getToUserId(jobClawUserId))
                .type(ResponseMessageType.TEXT)
                .content(content.toString())
                .passThrough(Map.of("aiCardId", aiCardId))
                .stream(true)
                .build();
    };
}
```

## 六、常见问题：深度解答

### 6.1 为什么需要 passThrough 透传参数？

**问题**：为什么不直接在消息模型中定义所有字段？

**答案**：

1. **避免模型膨胀**：不同通道的元数据差异很大，如果都定义在模型中，模型会无限膨胀
2. **保持解耦**：业务层无需理解通道特有的元数据
3. **灵活扩展**：新增通道时无需修改消息模型

**对比**：

```java
// 反例：直接定义所有字段
public class ChannelReceiveMessage {
    private String msgContentToken;      // 微信专用
    private String aiCardId;             // 钉钉专用
    private String openId;               // 飞书专用
    private String messageId;            // 飞书专用
    // ... 每新增一个通道，就要加几个字段
}

// 正例：使用 passThrough
public class ChannelReceiveMessage {
    private Map<String, Object> passThrough;  // ← 灵活扩展
}
```

### 6.2 事件总线会影响性能吗？

**问题**：异步事件处理会不会导致消息延迟？

**答案**：

1. **@Async 使用线程池执行**，不会阻塞通道层
2. **Spring 默认使用 `SimpleAsyncTaskExecutor`**，每个任务独立线程
3. **生产环境建议配置固定大小线程池**：

```yaml
spring:
  task:
    execution:
      pool:
        core-size: 10
        max-size: 50
        queue-capacity: 100
```

### 6.3 如何保证消息不丢失？

**问题**：异步处理时如果应用重启，消息会丢失吗？

**答案**：

1. **当前实现基于内存事件总线**，应用重启会丢失未处理事件
2. **生产环境建议**：
   - 使用消息队列（RabbitMQ/Kafka）替代 Spring 事件
   - 通道层在接收消息后立即持久化到数据库
   - 应用重启后从数据库恢复未处理消息
3. **当前架构的权衡**：
   - 优点：简单、易于实现、无额外依赖
   - 缺点：应用重启丢失消息
   - 适用场景：求职派当前是单体应用，消息丢失影响不大（用户可以重发）

## 七、小结：设计思想总结

消息模型与事件总线是求职派通道层与业务层解耦的核心基础设施，背后的设计思想是：

1. **统一抽象**：`ChannelReceiveMessage` 和 `ChannelResponseMessage` 屏蔽了渠道差异，业务层无需关心消息来自哪个渠道
2. **透传参数机制**：`passThrough` 灵活传递渠道特有元数据，避免消息模型无限膨胀
3. **事件驱动**：`ChannelEventPublisher` 实现通道层与业务层完全解耦，支持异步处理、多监听器、易于扩展
4. **异步非阻塞**：`@Async` + `@EventListener` 确保不阻塞 IM 连接，提升系统响应速度
5. **心跳机制**：通道层自己决定如何构建响应，后台任务无需关心通道特有的逻辑
6. **降级策略**：Agent 失败时回退到 LLM，保证用户始终能收到回复

**核心设计原则**：
- **关注点分离**：通道层负责消息适配，业务层负责业务逻辑
- **开闭原则**：对扩展开放（新增通道），对修改封闭（业务层无需改动）
- **单一职责**：每个组件只负责一件事（消息模型、事件发布、事件监听）

这种设计让求职派能够以统一的方式处理所有 IM 消息，同时保持通道层的灵活性和可扩展性。
