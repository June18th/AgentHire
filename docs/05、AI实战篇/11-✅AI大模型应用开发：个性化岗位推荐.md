# 一、订阅推荐设计
这一篇我们来实现一个基于用户个人偏好，并通过用户的自然语言描述，调用大模型后生成结构化 SQL  从校招派的数据库中获取匹配岗位信息，并推荐给用户。



一个真正的 AI 大模型应用开发案例。

## 1.功能设计
### 设计目的：
为了实现更精确的信息推荐，我们借助大模型来实现用户的意愿分析、并基于此做一些针对性的岗位推荐。

### 用户偏好录入&推荐流程：


在校招派的智能订阅体系中，整个推荐流程可以分为七个关键环节：  


首先，用户进入个人中心，通过自然语言描述自己的订阅偏好，比如：“想找一线城市互联网公司的产品经理岗位，优先推荐大厂校招。” 然后点击提交。



系统接收到请求后，前端会将用户的自然语言偏好数据封装为标准的 JSON 请求格式，并发送到后端服务。  


后端拿到数据后，先做一层校验，比如看内容是否为空、格式是否正确。通过验证后，系统会构造一个大模型调用请求，把用户的自然语言偏好和期望的结构化输出格式，一并发送给模型接口。  


大模型在收到请求后，会先理解语义，提取核心实体信息，比如城市（北京、上海等）、行业（互联网）、岗位（产品经理）、招聘类型（校招/实习）等，并把提取到的结果转成结构化对象返回。



后端解析模型返回的结果，生成对应的数据库查询逻辑（或调用数据库查询接口），比如一条类似这样的 SQL：



```sql
SELECT * FROM job_info
WHERE city IN ('北京','上海','广州','深圳')
  AND industry = '互联网'
  AND job_title LIKE '%产品经理%'
  AND recruit_type = '校招';
```



数据库执行查询后，返回符合偏好的岗位数据集，包括岗位 ID、企业名称、岗位详情、招聘截止日期等。  


最后，校招派后端会对这些岗位结果做进一步处理，比如计算匹配度、排序、补充岗位详情链接等，最终生成推荐列表，在用户的「我的推荐」模块中展示。

## 2.增值设计
上面那个属于**正向推荐方案**——用户主动录入偏好信息，系统根据输入去匹配岗位。而既然是“订阅”，那自然少不了“推送”能力：当有新的岗位上架时，系统应该能自动识别哪些用户对此类岗位感兴趣，并实时触达他们，从而真正实现**消息的及时推送**。



整个流程大致分为三步：



**①、管理员触发智能体采集，**管理员在校招派后台操作界面中，通过「校招派 Agent」模块启动采集流程。系统会自动生成任务 ID，随后由智能体接手完成从采集到发布的全流程处理。  


**②、智能体全链路数据处理，**这一步是整个推送体系的核心。

+ **任务分类**：智能体首先识别采集任务的类型——是图片、网页、链接还是纯文本，从而选择合适的采集方式；
+ **数据提取**：通过文本分析、Function Calling、MCP 等方式提取结构化校招信息；
+ **数据清洗**：利用预设规则对智能体返回的数据进行格式化、去噪、字段映射等处理；
+ **发布上线**：清洗完成且符合上线标准的数据会被自动发布到岗位库中。



**③、用户偏好匹配与消息推送，**当岗位数据发布上线后，系统会自动触发偏好匹配逻辑。

+ **偏好用户提取**：根据岗位属性（城市、岗位、行业、类型等），反向查询订阅了相关类别的用户；
+ **多渠道推送**：消息推送服务会根据用户的通知偏好优先级（例如：邮件 > 飞书 > 站内信），选择对应的渠道。



以邮件为例，系统会自动渲染 HTML 模板，包含岗位标题、详情链接、匹配度评分等信息，并通过 SMTP 协议完成发送，实现真正的“岗位上新即触达”。

# 二、订阅推荐实现
## 1.数据模型定义
如果从实际求职的角度来看，一个求职者想找岗位时，通常会有自己的一套筛选逻辑。比如说：

+ 想去什么公司？是不是得是 500 强、互联网大厂或者业内口碑好的那几家？
+ 岗位信息很关键——比如我是搞研发的，那我自然不想看那些销售、行政之类的职位。
+ 地点也要选，我是准备留在本地，还是想去北上广深闯一闯？
+ 招聘类型也得分清，是实习、校招，还是正式岗？
+ 薪资和福利当然也是重点，有没有五险一金、包住、转正机会……
+ 最后就是职位要求，看是不是对口、门槛高不高。  


而从系统设计的角度，我们在“岗位信息实体”里，其实已经覆盖了这些核心筛选条件：

+ 公司类型，比如央企、国企、民企这些；
+ 公司所属行业，比如数字能源、新能源、互联网、教育等；
+ 工作地点，比如北京、武汉，或者全国可选；
+ 招聘类型，像校招、春招、秋招、实习；
+ 招聘对象，比如 2025 届、2026 届毕业生；
+ 岗位类别，比如自动驾驶算法、AI 数据工程、软件开发、嵌入式开发等等。



所以我们希望大模型可以帮我从用户的描述中提取以下信息用于筛选推荐。

```java
public record UserInterestBo(
        /**
         * 公司类型
         */
        @JsonPropertyDescription("公司类型：如央企、国企、民企")
        String companyType,

        /**
         * 公司行业
         */
        @JsonPropertyDescription("公司所属的行业类型：如数字能源/新能源, 工程承包/建筑施工, 科技, 互联网, 教培, 教育, 学校等")
        String companyIndustry,

        /**
         * 工作地点
         */
        @JsonPropertyDescription("工作地点，通常为地址，如：武汉、北京、全国")
        String jobLocation,

        /**
         * 招聘类型
         */
        @JsonPropertyDescription("招聘类型：如校招、秋招、春招、实习")
        String recruitmentType,
        /**
         * 招聘对象
         */
        @JsonPropertyDescription("招聘对象：如2026年毕业生, 2025年毕业生")
        String recruitmentTarget,
        /**
         * 岗位
         */
        @JsonPropertyDescription("岗位，如自动驾驶算法 AI数据工程 AI安全/隐私保护 大模型应用 通用软件开发 嵌入式软件开发 操作系统与编译器开发 数据")
        String position) {
}

```

基于上面这套结构化的岗位数据，我们就可以做用户偏好匹配了。



当然，直接每次都去调用大模型来解析用户的自然语言偏好，这种做法虽然“智能”，但从实际落地来看，问题不少——  


一是**响应慢**，模型接口调用往往有延迟；

二是**容易限速**，尤其是多人同时订阅、批量触发时；

三嘛，**太贵了**，每次都让模型干同样的事确实有点浪费。  


所以，我们选择定义一个**用户偏好实体（UserInterestEntity）**，来持久化用户的订阅意图。



这个实体的结构其实和上面岗位信息的定义几乎一致，只是**多了用户关联字段**（比如 user_id 等）。



这样做的好处是：



+ 用户第一次设置偏好时，通过大模型解析；
+ 后续推荐或推送时，直接使用数据库中缓存的结构化偏好；
+ 既节省算力，也能保证查询的速度和一致性。



简单说，就是——**模型负责第一次理解，人负责之后的高效利用。**

```java
@Data
@Accessors(chain = true)
@Entity(name = "user_interest")
public class UserInterestEntity {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(name = "user_id")
    private Long userId;

    /**
     * 兴趣
     */
    @Column(name = "interest")
    private String interest;
    /**
     * 公司类型
     */
    @Column(name = "company_type")
    private String companyType;

    /**
     * 公司行业
     */
    @Column(name = "company_industry")
    private String companyIndustry;

    /**
     * 工作地点
     */
    @Column(name = "job_location")
    private String jobLocation;

    /**
     * 招聘类型
     */
    @Column(name = "recruitment_type")
    private String recruitmentType;
    /**
     * 招聘对象
     */
    @Column(name = "recruitment_target")
    private String recruitmentTarget;
    /**
     * 岗位
     */
    @Column(name = "position")
    private String position;

    /**
     * 创建时间
     */
    @Column(name = "create_time")
    private Date createTime;


    @Column(name = "update_time")
    private Date updateTime;
}
```

## 2.基于大模型提取结构化的偏好设置
接下来是核心，当用户提交一段自然语言描述的文本，我们要基于这个文本生成结构化的数据，如输入：

```java
我是2026年毕业的大学生，希望寻找一些服务器开发、前端开发、算法相关的岗位，希望在北京、上海、深圳工作
```

预期输出：

```java
行业偏好：互联网

工作地点：北京,上海,深圳

招聘对象：2026年毕业生

目标职位：服务器开发/前端开发/算法
```



实现起来不难，毕竟这属于大模型的基本使用；在现有的AI封装入口中，添加一个获取`ChatClient`的方法，后续大模型的交互，我们就都交给它来实现。

```java
/**
 * 提交用户的偏好信息
 *
 * @param text
 */
public void submitInterest(String text) {
    ChatClient chatClient = this.aiModelFacade.getMainChatClient();
    // 根据用户传入的信息，提取需要关注的内容
    UserInterestBo userInterestBo = chatClient.prompt()
            .user("我现在是一个准备找工作的人，下面是我的个人信息，我希望从中提取出用于筛选招聘岗位的关键信息: \n" + text)
            .call().entity(UserInterestBo.class);
}
```

## 3.订阅偏好持久化
到这里，大模型已经能帮我们把用户的偏好转成结构化数据了，但问题来了——这些数据**还不能直接用**。  


为什么？



主要是因为**大模型提取出来的字段值，可能和我们系统的字典库不完全一致。**  


举个例子：

+ 我们在校招派的字典里，公司类型存的是“民企”，结果模型给我们提了个“私企”，那数据库一查，直接对不上；
+ 又比如招聘类型，我们的字典里是“暑期实习”，模型返回“暑假实习”，看起来差不多，但系统可不认这个同义词。  


这些“看似正确但不匹配”的值，在真实查询场景中会让整个推荐逻辑直接失效。  


所以，为了保证后续的推荐准确性，我们需要对这些字段值进行**二次清洗与映射**。简单说，就是把模型生成的结果重新“对齐”到系统内部的标准字典。  


这部分工作其实可以直接**沿用后台发布草稿数据时的清洗流程**，让模型提取的偏好信息经过统一转换，再入库使用。  


这样一来，系统数据就干净统一，后面的查询、推荐、推送逻辑才能稳稳跑起来。

## 4.订阅偏好查询
接下来就是按个人偏好去做“主动查询”的接口了。



本质上这一步没什么玄学，就是一条老老实实的 SQL，把用户的订阅偏好翻译成查询条件丢给数据库就行。



需要注意的地方主要有两个：



第一，订阅偏好里很多字段是多值的，比如工作地点有人会同时勾北京、上海、广州、深圳，这种要先把条件拆开，统一转成 in 查询或等价的表达；



第二，别把用户原始偏好直接拼 SQL，先过一层标准化映射，确保都落在系统字典里，不然命中率感人。



实现上我把这套逻辑放在 UserInterestService 里，专门根据用户的订阅偏好组装查询参数，请求体定义成 UserInterestRecommendReq，包含城市、行业、公司类型、岗位类别、招聘类型、毕业年份这些核心维度。这样接口就干净了：入口拿到 UserInterestRecommendReq，内部做一次条件拆分与规范化，最后生成稳定可复用的查询语句，推荐结果就能稳稳当当地跑出来。

```java
/**
 * 用户偏好推荐请求
 *
 * @author YiHui
 * @date 2025/8/22
 */
@Data
public class UserInterestRecommendReq extends PageReq {
    /**
     * 公司类型
     */
    private List<String> companyTypeList;
    /**
     * 公司行业
     */
    private List<String> companyIndustryList;
    /**
     * 工作地点
     */
    private List<String> jobLocationList;
    /**
     * 招聘类型
     */
    private List<String> recruitmentTypeList;
    /**
     * 岗位
     */
    private List<String> positionList;
    /**
     * 招聘对象
     */
    private String recruitmentTarget;
}
```

# 三、小结
这篇文章算是一个“小而完整”的实战案例，主要想展示**如何把大模型的能力真正融入到业务功能中**，让它成为系统里的一个基础单元去参与工作。  


整体的开发流程其实和我们日常写的 CURD 业务差别并不大，依旧是那套“数据流 + 逻辑流”的思路，只是多了一个智能化环节——让大模型帮我们做信息理解与结构化。
