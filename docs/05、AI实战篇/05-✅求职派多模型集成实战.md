# 05-✅求职派多模型集成实战

在集成了讯飞大模型后，求职派至少有两个大模型可供使用了。上一篇演示的使用方式比较初级，通过硬编码的方式来选择具体的大模型，这种方式显然不太适合一个生产级别的项目。


接下来我们看一下，如何实现多模型的快速切换。

## 一、多模型集成实现
### 1.模型调用封装
在求职派中，使用大模型的入口主要在 `GatherAiAgent`类中。


![](https://cdn.nlark.com/yuque/0/2025/png/35158118/1753835069471-4650ea02-de5f-40da-a3d2-18608791bcba.png)

如果只有一个模型，当然可以将ChatClient/ChatModel 直接注入到这个Agent中使用；但是如果想用多模型，这样就不合适了。


所以我提取了一个 `AiModelFacade` 门面类，来作为模型调用的交换机；将模型相关的注入、选择、切换等逻辑，全部封装在里面，实现对上层业务的无感知。


### 2.AiModelFacade实现
接下来，我们需要重点实现的，就在这个类中。实现方案多样，最简单就是面条式编码方式。

```java
@Component
public class AiModelFacade {
    public static final String SYSTEM_PROMPT = """
    你现在是一个专业的数据挖掘者，可以从我提供给你的文本内容、表格文件、html文本中获取用户希望的信息；
    如果我给你的是一个http链接，则借助function tool crawlerHttpTable从链接对应的网页中找到表格元素返回给用户希望的信息
    """;
    private static final String IMG_MODEL = "GLM-4V-Flash";
    private static final String CHAT_MODEL = "GLM-4-Flash";

    /**
     * 文本类大模型
     */
    private final ChatClient chatClient;

    /**
     * 图片视觉理解的模型
     */
    private final ChatClient imgChatClient;

    private final ChatModel chatModel;

    private final SparkLiteModel sparkLiteModel;

    /**
     * fixme 待集成多个模型
     *
     * @param chatModel
     */
    public AiModelFacade(ZhiPuAiChatModel chatModel, SparkLiteModel sparkLiteModel) {
        this.chatModel = chatModel;
        this.sparkLiteModel = sparkLiteModel;

        chatClient = ChatClient.builder(chatModel)
        .defaultSystem(SYSTEM_PROMPT)
        .defaultOptions(ChatOptions.builder().stopSequences(Collections.emptyList()).build()) // 取消默认停止符
        .defaultAdvisors(new SimpleLoggerAdvisor())
        .build();

        // 图片理解
        imgChatClient = ChatClient.builder(chatModel)
        .defaultSystem(SYSTEM_PROMPT)
        .defaultOptions(ChatOptions.builder()
                        .model(IMG_MODEL)
                        .stopSequences(Collections.emptyList()).build())
        .defaultAdvisors(new SimpleLoggerAdvisor())
        .build();
    }

    public ChatClient getChatClient() {
        return chatClient;
    }

    public ChatClient getImgChatClient() {
        return imgChatClient;
    }


    public ChatModel getChatModel() {
        return chatModel;
    }

    public ChatClient getSparkClient() {
        return ChatClient.builder(sparkLiteModel)
        .defaultSystem(SYSTEM_PROMPT)
        .defaultOptions(ChatOptions.builder().stopSequences(Collections.emptyList()).build()) // 取消默认停止符
        .defaultAdvisors(new SimpleLoggerAdvisor())
        .build();
    }
}
```


实现方式非常直观，也很容易理解，来一个 `ChatModel`就注入一个实例，然后提供对应的 `ChatClient`方法就行了；接下来就是封装一个方法，根据传参来选择具体执行的Model。


定义一个大模型枚举，用于模型选择的传参。

```java
@Getter
public enum GatherModelEnum implements StringBaseEnum {
    ZHIPU("ZhiPu", "智谱"),
    SPARK_LITE("SparkLite", "讯飞星火"),
    CHAT_GPT("ChatGPT", "ChatGPT"),
    DEEPSEEK("DeepSeek", "DeepSeek"),
    ;
    private final String value;
    private final String desc;

    GatherModelEnum(String value, String desc) {
        this.value = value;
        this.desc = desc;
    }
}
```

然后实现一个基于传参返回的 `ChatClient` 或者 `ChatModel`，简单的 switch 即可。

```java
public ChatClient chatClient(GatherModelEnum model) {
    return switch (model) {
        case SPARK_LITE -> getSparkClient();
        case ZHIPU -> getChatClient();
        default -> getChatClient();
    };
}

public ChatModel chatModel(GatherModelEnum model) {
    return switch (model) {
        case ZHIPU -> chatModel;
        case SPARK_LITE -> sparkLiteModel;
        default -> getChatModel();
    };
}
```


相应的，我们需要调整一下访问入口，新增一个传参。因为在进行替换的时候，虽然用的是同一个厂商的大模型，但是文本模型和图片模型的底层选择还是不一样的。

## 二、多模型实现层优化
### 1.优化分析
#### 问题一：模型的手动注入
来看一下，我们实现的 `AiModelFacade` 有什么问题？


![](https://cdn.nlark.com/yuque/0/2025/png/35158118/1753839034086-1bcd8b4e-098e-4244-a9cb-b97e5586a906.png)

从初始化可以看出，每添加一个模型，就需要主动注入到这个类中；也就意味着这里需要同步进行修改。

#### 优化一：自动识别新增的模型
针对这个问题，优化方案是希望新增的模型可以自动注入，不需要手动注入，比如直接在传参处注入 `List<ChatModel>`。


![](https://cdn.nlark.com/yuque/0/2025/png/35158118/1753839493606-0e4e9e01-177b-4ed6-a866-a0554c75dfaa.png)


![](https://cdn.nlark.com/yuque/0/2025/png/35158118/1753839531261-9d3e7f52-77bc-4389-a660-2ba4454ab677.png)

通过IDEA Spring定义可以看到注入了两

#### 问题二：模型选择硬编码
其次就是通过枚举来选择模型这里，同样需要进行优化，以适配不同的模型选择；再然后就是modelName的选择，因为现阶段只有智谱支持图文，所以只实现了智谱这一套，显然并不利于后续的扩展。


![](https://cdn.nlark.com/yuque/0/2025/png/35158118/1753839108452-b2228d69-4036-46c5-80fe-5745579c2e9d.png)

#### 优化二：硬编码改条件匹配
硬编码的方式会导致每次新增模型就需要适配；能想到的改进方案就是给每个模型扩展一个方法，接收一个 `GatherModelEnum`传参，如果返回 true，表示用这个模型；否则表示不用这个模型。


从而将硬编码的逻辑从 Facade 中迁移到集成的模型内，这样模型的新增都只需要改它就行了。


### 2.完整优化方案
#### 步骤一：统一接口抽象定义
为了实现这个场景，我们通过定义一个统一的接口，来定义模型的行为。

```java
/**
 * 求职派大模型集成的统一接口
 *
 * @author YiHui
 * @date 2025/7/30
 */
public interface OcChatModelApi {
    String GATHER_SYSTEM_PROMPT = """
            你现在是一个专业的数据挖掘者，可以从我提供给你的文本内容、表格文件、html文本中获取用户希望的信息；
            如果我给你的是一个http链接，则借助function tool crawlerHttpTable从链接对应的网页中找到表格元素返回给用户希望的信息
             """;

    /**
     * 根据传参获取对应de
     *
     * @param req
     * @return
     */
    ChatClient chatClient(ModelSelectReq req);

    /**
     * 获取模型
     *
     * @return left: 大模型  right: 模型名
     */
    Pair<ChatModel, String> model(ModelSelectReq req);

    /**
     * 当前模型对应的枚举定义，用于模型选择切换
     *
     * @return
     */
    GatherModelEnum modelEnum();
}

```


然后每个模型，完成上面接口的实现；为了简化实现，我们先基于上面的接口实现一个抽象类，

```java
/**
 * 求职派大模型集成的统一接口
 *
 * @author YiHui
 * @date 2025/7/30
 */
public abstract class AbsOcChatModelApi implements OcChatModelApi {
    /**
     * 返回文本聊天的大模型客户端
     *
     * @return
     */
    public abstract ChatClient chatClient();

    /**
     * 返回图文聊天的大模型客户端
     *
     * @return
     */
    public ChatClient imgClient() {
        throw new BizException(StatusEnum.MODEL_NOT_SUPPORT);
    }

    /**
     * 获取大模型
     *
     * @return
     */
    public abstract ChatModel chatModel();

    /**
     * 获取文本大模型名称
     *
     * @return
     */
    public abstract String chatModelName();

    /**
     * 获取图片理解大模型名称
     *
     * @return
     */
    public String imgModelName() {
        throw new BizException(StatusEnum.MODEL_NOT_SUPPORT);
    }


    /**
     * 根据传参获取对应de
     *
     * @param req
     * @return
     */
    public ChatClient chatClient(ModelSelectReq req) {
        if (req.mode() != modelEnum()) {
            return null;
        }
        return req.type() == GatherModelTypeEnum.IMAGE_MODEL ? imgClient() : chatClient();
    }

    /**
     * 获取模型
     *
     * @return left: 大模型  right: 模型名
     */
    public Pair<ChatModel, String> model(ModelSelectReq req) {
        if (req.mode() != modelEnum()) {
            return null;
        }
        String modelName = req.type() == GatherModelTypeEnum.IMAGE_MODEL ? imgModelName() : chatModelName();
        return Pair.of(chatModel(), modelName);
    }
}

```

#### 步骤三：Facade使用层的优化
```java
@Component
public class AiModelFacade {

    private final List<OcChatModelApi> list;

    /**
     * 注入
     *
     * @param list
     */
    public AiModelFacade(List<OcChatModelApi> list) {
        this.list = list;
    }

    public ChatClient chatClient(ModelSelectReq req) {
        for (OcChatModelApi api : list) {
            ChatClient client = api.chatClient(req);
            if (client != null) {
                return client;
            }
        }
        throw new BizException(StatusEnum.MODEL_MISMATCH_SUPPORT);
    }

    public Pair<ChatModel, String> model(ModelSelectReq req) {
        for (OcChatModelApi api : list) {
            Pair<ChatModel, String> model = api.model(req);
            if (model != null) {
                return model;
            }
        }
        throw new BizException(StatusEnum.MODEL_MISMATCH_SUPPORT);
    }
}
```
