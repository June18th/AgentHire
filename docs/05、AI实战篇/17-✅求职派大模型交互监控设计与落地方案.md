# 求职派大模型交互监控设计与落地方案

> **文档说明**：本文记录 JobClaw 在 `core` 与 `agent` 体系中的大模型调用监控方案，包含统计口径、数据模型、接口权限、页面展示和隐私策略。  
> **适用范围**：`core`、`agents`、`app` 三层调用链，以及管理员与普通用户的可见界面。  
> **状态**：已落地实施。

---

## 一、目标

JobClaw 的大模型调用分布在多个位置：

- `core` 中的意图识别、会话总结、身份提取、默认对话
- `agents` 中的职位抽取、推荐、文本/图像理解
- ReAct 与工具调用内部的多轮真实模型请求

此前的问题是：

- 只能看到零散日志，无法统一统计
- 无法区分“一次业务调用”和“多次真实模型请求”
- 没有用户维度的使用明细
- 没有统一费用估算
- 监控数据无法直接在管理端查看

本方案的目标是建立一套统一、低侵入、可扩展的监控体系：

- 覆盖 `agent` 与 `core` 中的 ChatModel 调用
- 同时记录逻辑调用与物理请求
- 支持管理员全局监控
- 支持用户查看自己的调用明细
- 支持 Token、耗时、费用、成功率等统计
- 支持脱敏采样与隐私保护

---

## 二、监控模型

### 2.1 两层模型

监控分成两层：

- **Invocation**：一次业务语义上的调用
- **Request**：一次真实的大模型请求

示例：

- 用户发出一句消息，进入 `MsgRouter`
- 该消息经过意图识别，命中某个 Agent
- Agent 可能在 ReAct 中连续调用多次模型
- 对监控而言，这属于 1 次 Invocation + N 次 Request

### 2.2 统计口径

建议统一统计以下指标：

- 调用次数
- 成功率 / 失败率 / 取消率
- 平均耗时、P95 耗时
- 输入 Token、输出 Token、总 Token
- 预估费用
- 按 Agent、用途、模型、Provider、模式聚合

### 2.3 维度定义

建议统一携带以下上下文字段：

- `invocationId`
- `jobClawUserId`
- `conversationId`
- `channel`
- `agent`
- `operation`
- `mode`
- `provider`
- `model`
- `modelType`
- `outcome`

其中：

- `agent` 表示业务 Agent 名称
- `operation` 表示调用用途，如 `intent_classification`
- `mode` 表示 `SYNC` / `STREAM`
- `outcome` 表示 `SUCCESS` / `FAILED` / `CANCELLED`

## 三、实现思路

### 3.1 `core` 统一接入

核心做法是把观测能力放在 `ModelProviders` 与 `LlmCaller` 之间：

- `ModelProviders` 负责返回模型实例
- `LlmCaller` 负责构建调用上下文
- `DefaultLlmMonitor` 负责发布监控事件

这样可以覆盖：

- 普通 `ChatClient` 调用
- `BizAgentLlmCaller`
- `SimpleLlmCaller`
- `UserPreferenceBasedLlmCaller`
- `JobLlmCaller`
- ReAct 内部直接模型调用

### 3.2 统一调用上下文

在调用开始时构建 `LlmCallContext`：

- 如果来自具体业务 Agent，就带上 `agent`
- 如果来自非业务路径，就使用默认 `operation`
- 对流式调用与同步调用分别标记 `mode`

### 3.3 ReAct 内部请求统计

ReAct 中的内部推理会绕过普通 `ChatClient` 的上层调用链，因此需要单独包一层：

- 每次内部 `chatModel.call(...)` 记为一个 Request
- 这些 Request 共享同一个 Invocation
- 这样既能看见单次业务耗时，也能看见内部循环次数

### 3.4 事件驱动持久化

监控数据通过 Spring 事件发布：

- `core` 发布中立监控事件
- `app` 模块监听事件并落库
- 持久化失败不影响主调用

这是一个很重要的边界：

- `core` 不依赖 Web 层
- `core` 不直接依赖 JPA Repository
- `app` 负责最终的数据服务和页面展示

## 四、数据模型

### 4.1 `llm_invocation`

逻辑调用汇总表，记录一次业务调用的总览信息：

- `id`
- `job_claw_user_id`
- `conversation_id`
- `channel`
- `agent`
- `operation`
- `mode`
- `outcome`
- `duration_ms`
- `request_count`
- `input_tokens`
- `output_tokens`
- `total_tokens`
- `estimated_cost`
- `error_message`
- `create_time`

### 4.2 `llm_request`

真实请求明细表，记录一次模型请求：

- `id`
- `invocation_id`
- `job_claw_user_id`
- `conversation_id`
- `channel`
- `agent`
- `operation`
- `mode`
- `provider`
- `model`
- `model_type`
- `outcome`
- `duration_ms`
- `input_tokens`
- `output_tokens`
- `total_tokens`
- `estimated_cost`
- `error_message`
- `prompt_sample`
- `response_sample`
- `create_time`

### 4.3 费用计算

费用按模型配置中的单价估算：

- `input-price-per-million-tokens`
- `output-price-per-million-tokens`

建议规则：

- 只有在拿到 Usage 且有价格配置时才计算费用
- 无 Usage 时费用为空
- 未配置价格时费用为空
- 费用字段是“预估值”，不代表账单金额

### 4.4 采样与脱敏

建议默认策略：

- 成功请求采样保存 1%
- 失败请求采样保存 100%
- 采样内容仅管理员可见

脱敏内容建议覆盖：

- 邮箱
- 手机号
- 身份证号
- API Key
- Bearer Token

并且：

- Prompt / Response 要截断保存
- 媒体不保存原文，只记录类型、数量、大小

## 五、指标体系

建议通过 Micrometer / Actuator 暴露以下指标：

- `jobclaw.llm.invocations`
- `jobclaw.llm.requests`
- `jobclaw.llm.request.duration`
- `jobclaw.llm.tokens`
- `jobclaw.llm.estimated.cost`
- `jobclaw.llm.telemetry.dropped`

### 5.1 标签约束

指标标签只允许低基数维度：

- `agent`
- `operation`
- `provider`
- `model`
- `mode`
- `outcome`

不要把以下字段放进 Metrics 标签：

- 用户 ID
- 会话 ID
- 消息 ID
- Request ID

这些字段适合放进数据库，不适合放进 Metrics。

## 六、接口设计

### 6.1 管理员接口

建议提供：

- `GET /api/admin/llm-monitor/overview`
- `GET /api/admin/llm-monitor/trends`
- `GET /api/admin/llm-monitor/breakdown`
- `GET /api/admin/llm-monitor/calls`
- `GET /api/admin/llm-monitor/calls/{id}`

用途：

- 全局用量概览
- 趋势图
- 维度分布
- 调用列表
- 调用详情与请求链路

### 6.2 用户接口

建议提供：

- `GET /api/user/llm-usage/overview`
- `GET /api/user/llm-usage/trends`
- `GET /api/user/llm-usage/calls`
- `GET /api/user/llm-usage/calls/{id}`

用户侧规则：

- 从 `ReqInfoContext` 取当前登录用户
- 服务层强制按 `jobClawUserId` 过滤
- 不接受客户端传入 userId 覆盖
- 查询不到自己的记录时直接返回不存在

### 6.3 权限边界

权限建议如下：

- 管理员可以看全局数据和采样内容
- 普通用户只能看自己的调用明细
- 未绑定站内账号的调用，只允许管理员查看

## 七、前端展示

### 7.1 管理员页面

建议新增 `/admin/llm-monitor`：

- 调用量概览
- 成功率
- P95 延迟
- Token 总量
- 费用总量
- Agent / 模型 / 操作维度分布
- 调用明细表格

### 7.2 用户页面

建议在个人中心新增“模型用量”菜单：

- 我的调用次数
- 我的成功率
- 我的平均耗时
- 我的 Token 与费用
- 我的调用明细

用户页面只展示元数据，不展示敏感内容。

## 八、落地边界

### 8.1 已纳入的调用范围

当前方案覆盖：

- `core` 中通过 `ModelProviders` 创建的模型
- `SimpleLlmCaller`
- `BizAgentLlmCaller`
- `UserPreferenceBasedLlmCaller`
- `JobLlmCaller`
- ReAct 内部直接模型调用

### 8.2 暂不纳入的范围

先不强行覆盖：

- `app/gather` 中旧式、绕过 `ModelProviders` 的遗留模型调用
- 图像、语音、Embedding 的全部统一明细分析

这些后续可以按同一套模型逐步接入。

---

## 九、验证计划

建议验证这些场景：

- 同步调用是否有 Invocation 和 Request 两层记录
- 流式调用是否正确收尾
- ReAct 是否能记住多次内部请求
- Token / 费用是否正确汇总
- 无 Usage 时费用是否为空
- 采样与脱敏是否生效
- 管理员是否可看全局和采样
- 普通用户是否只能看自己的数据
- 越权访问是否被拦截
- Metrics 是否没有高基数标签

---

## 十、备注

这套方案的关键不是“多打一层日志”，而是把调用语义、请求链路、用户归属和费用估算放进同一个体系里。

这样以后我们可以很自然地回答几个问题：

- 哪个 Agent 最耗 Token
- 哪个用户最常调用
- 哪个模型最贵
- 哪类任务失败最多
- 一次业务调用内部到底跑了几轮模型

这才算是真正可用的监控。
