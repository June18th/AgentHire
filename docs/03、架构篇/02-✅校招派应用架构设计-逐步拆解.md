接下来我将从整体的架构设计出发，给大家介绍一下校招派整体的设计，以及基于SpringAI来构建智能体的强大能力（姑且称校招派为一个校招信息采集展示智能体吧😀）



在校招派的实现中，基于Spring AI实现了校招信息录入、采集、保存、数据清洗、信息展示等能力，核心表现为：

+ 用户提交需要采集的目标（支持文本数据、html网页、http链接、csv文件、excel文件、图片六大格式）
+ 将用户提交的任务放入池子，由统一的任务调度来驱动SpringAI实现任务的执行
+ SpringAI实现与大模型的对话，实现采集任务
+ 支持多轮连续对话：针对大模型一次无法完整返回结果的情况，实现了自动对话，让大模型接着上次的回答内容继续采集任务
+ 提供了工具调用(Function Tools)，让大模型可以获取http链接对应的内容
+ 支持多模态的调用，实现与大模型之间的图文交互

## 三、多轮对话
在实现的过程中，我发现采集任务的信息要素比较多时，大模型并不能保证一次就将所有的数据返回，相信大家在使用 AI 工具的时候也碰到过类似的情况；这时候，就需要我们告知模型，接着上次返回的内容，继续后续的内容生成给我们。



由于大模型本身是无状态的，当我们希望大模型可以接着上一次对话继续返回结果时，就需要我们将上次的对话上下文也一并返回给大模型，于是我们需要实现聊天记忆来支持多轮对话的能力；这里，我们借助了 SpringAI 的 `ChatMemory`来实现。

## 四、使用Function Call
出于成本的考虑，我们优选的大模型为智普清言，这样大家在学习这个项目的时候也不需要额外的花钱啦，ps：大模型的使用成本可不算低~



既然是免费的大模型，那么它的能力必然不是第一梯队的，比如我给它http链接，希望它能帮我抓取对应的网页内容进行解析，他就很为难；既然大模型也有手不够长的时候，那就不妨让我们通过Function Call给其添加一些“人形外挂”。



所以大家会看到，在校招派中，我们设计了一个基于爬虫的http链接解析工具（后续会再扩展一个基于浏览器的链接解析工具，用于解决目标网站是由js渲染的场景）

## 五、结构化返回

SpringAI当然也考虑到了，它设计了一套基于<font style="color:rgb(111, 66, 193);background-color:rgb(246, 248, 250);">StructuredOutputConverter</font> 的结构化返回方案，在和大模型进行交互时，给他插入一个定义了返回格式的JSON Schema 就可以了。

## 七、任务池调度
在项目开始前，我并没有考虑任务池的调度模式，这个完全是事后的倒逼行为。



在原型跑通之后，我发现大模型的响应是真的慢。而且免费的大模型还有一个恶心的限制，那就是访问频率的限定，不支持高并发（>3的请求都算高并发？）



所以为了更好的交互体验，基于任务池的调度方案是必须考虑的一件事，于是我做了一个非常简单的基于db的异步任务调度方案，所以我们的架构叕一次升级了
