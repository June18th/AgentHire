# 01-✅求职派业务架构设计-逐步拆解

业务架构主要用于阐述求职派的业务策略、关键的业务过程。更多的是表明这个产品的核心价值、业务流程。

## 一、业务流程
求职派的原始需求来自于 [https://paicoding.com/article/detail/2519100025427969](https://paicoding.com/article/detail/2519100025427969) ，基于这个目标，最开始我对求职派进行了一个简单功能拆解

整体内容区分三块：

1. 用户：登录  + 会员充值
2. 校招信息展示（前台、后台管理页）
3. 校招信息采集（基于SpringAI + 大模型实现校招信息录入）

### 2.业务架构图
针对上面的功能模块拆解，结合具体的实施策略，针对上面拆分的五大模块又做了一个调整，主要是校招信息维护这里，目前更多的还是采用人力维护的方式，技术含量不太大（不好吹🐂🍺）所以就拿掉，然后将SpringAI大模型这块，拆分成两部分：大模型 + 任务调度。


因此整个校招牌的业务功能，就划分新的五大模块

1. 实现登录、鉴权的 **用户认证与权限管理** 模块
2. 基于SpringAI实现的大模型能力应用层 **校招信息采集Agent**
3. 支撑SpringAI采集任务的 **异步AI调度任务 **模块
4. 以微信支付为基座实现的 **会员充值 **模块
5. 统一维护管理全局字典、业务配置的 **全局业务字典管理 **模块

#### 2.1. 用户认证与权限管理
1. 支持关注微信公众号，发送验证码登录的方式。
2. 基于 jwt（JSON Web Token）的用户登录鉴权，实现无状态的身份验证，提高系统的可扩展性。同时，自定义的权限管理模块，能精准控制不同用户的操作权限。


按照业务架构的分层视图，我们将用户认证/权限管理这块内容补全，对应的业务架构图为：


![](https://cdn.nlark.com/yuque/0/2025/png/35158118/1753772223766-fec0945c-4fd2-4a4b-a33d-30c1e98b4d18.png)

#### 2.2 基于大模型的校招信息采集Agent
借助 SpringAI 实现大模型的数据采集，这是求职派的一大技术亮点。

1. 提交采集任务后，系统会在任务表中插入一条记录，然后在后台异步执行采集任务，不影响前端用户的操作
2. 支持多种数据输入方式，包括文本、图片、csv 文件、excel 文件、http 网页、http 链接等，能全面覆盖各类校招信息的来源，满足多样化的采集需求。


然后我们的业务架构图中，分别在应用层和领域层中添加 `任务录入 + gather领域`，将采集的结果映射为 `oc领域`的 `OcDraftEntity实体`


![](https://cdn.nlark.com/yuque/0/2025/png/35158118/1753772646994-09ca701e-ae91-4992-840b-d8789f652697.png)


#### 2.3 异步AI调度任务机制
我们设计了一个任务池的数据模型，将用户提交的采集任务先下放到任务池，然后进行异步的任务调度执行，从而减少大模型响应慢的问题；因此我们可以在上面的业务架构图中，添加任务管理相关的内容即可

集成微信支付实现会员充值功能，所以在上面的业务架构图中，就会新增微信支付、会员充值的相关内容


![](https://cdn.nlark.com/yuque/0/2025/png/35158118/1753773757103-033908ec-c932-454e-a562-db0796d854bf.png)

#### 2.5 全局字典
统一维护全栈的配置信息，便于我们对系统进行配置管理和维护；设计全局字典的主要目的就是前后台进行交互时，会有很多预定义的字段信息，比如公司类型、招聘对象等，如果全部用硬编码方式，则前后端都需要各自维护一套，很容易导致映射不上的情况。


而且这些数据字段的取值，也没办法完全确定永远是哪几个，必然会遇到修改的情况，如果每次修改都需要同步改造前后端的相关代码，这个成本就有点高


因此我们设计了全局字典进行统一维护，前端也直接从字典中获取常量信息，这样就可以实现实时更新了


![](https://cdn.nlark.com/yuque/0/2025/png/35158118/1753774099447-a56a1dbc-dee3-4722-9770-a6d31658e513.png)

## 三、小结
这一篇的设计，主要是从功能角度出发，通过主业务流程，拆解出求职派的核心功能模块的思维导图，然后再基于这个思维导图得到核心的五大功能模块；然后从上到下，由用户接入层到下面的应用、领域层进行功能单元的拆解，最终形成我们的业务架构图。


那怎么评判这个业务架构图的效果呢？


如果基于这张图，你能知道求职派的核心功能模块、基于它可以抓住一些关键的技术路线，那么就可以说它是合格的。如果对这张业务架构图有什么疑问，欢迎评论交流。
