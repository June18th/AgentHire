# 04-✅求职派表结构设计

求职派的表结构非常简单，我们快速过一遍。

## 一、数据表
项目源码中提供了完整的数据库表 Schema，在 `resources/db/changelog` 目录下可自取。


![](https://cdn.nlark.com/yuque/0/2025/png/35158118/1753778713139-7ff87860-1747-46dd-b47e-8d3261e945f6.png)


当前核心业务表包括如下几类：

### 1.用户数据表
定义用户的基本信息，包含微信id（用于登录与个人身份识别，这个也是mcp server鉴权code，不要泄露）


![](https://cdn.nlark.com/yuque/0/2025/png/35158118/1753778828188-1808c99f-7efb-44c0-a6b4-c3b9566c8be9.png)

### 2.充值数据表
记录用户的充值明细，其中会定义与微信支付的相关数据


![](https://cdn.nlark.com/yuque/0/2025/png/35158118/1753778928074-7b83f967-0d4a-4b6b-b5d3-8a2938db1deb.png)

### 3.全局字典表
简单的kv数据模型，用于存储站点定义的字典信息；注意 dict_key + dict_value 要求唯一限定；不过我们没有做强管控。


![](https://cdn.nlark.com/yuque/0/2025/png/35158118/1753778967352-ce5e124c-5a37-47f5-a728-f5649e3d04fa.png)

为了简化使用，在项目初始化时，求职派就已经自动将项目中的所有枚举定义导入了全局字典表，对应的 SQL 文件在 `resources/db/changelog/init_data_250722.sql`


![](https://cdn.nlark.com/yuque/0/2025/png/35158118/1753779115885-0e7e2795-309e-42ce-a711-0f951c71f4f3.png)

### 4.任务表
求职派将用户提交的任务做了异步处理，即提交的任务先进 `gather_task` 任务池，因此这个表主要用于存储用户提交的任务上下文


![](https://cdn.nlark.com/yuque/0/2025/png/35158118/1753779214643-b439b741-d023-4ae6-9366-19235739fb96.png)


### 5.草稿数据
由于无法完全信赖大模型的返回（总会有一些识别异常的情况），因此大模型的输出都是先存草稿数据，编辑后再推送到正式库


![](https://cdn.nlark.com/yuque/0/2025/png/35158118/1753779300407-dca535d1-f484-447f-88a3-f907a9b0ec78.png)

### 6.职位信息表
这里存储的是正式可以给用户使用的职位信息，基本数据模型与草稿数据差不多，区别在于这里的数据都是经过人工审核处理的


![](https://cdn.nlark.com/yuque/0/2025/png/35158118/1753779364375-35b9991a-9127-42f0-abf3-dedc9a25761f.png)

### 7.投递记录表

为了把“看岗位”推进到“管投递”，当前新增了个人投递记录相关表。迁移脚本位于：

```text
backend/src/main/resources/db/changelog/job_application_260705.sql
backend/src/main/resources/db/changelog/job_application_company_type_260706.sql
```

核心表包括：

- `job_application`：用户投递记录主表，保存公司、岗位、投递链接、公司类型、当前状态、优先级、截止时间、投递时间、下次跟进时间和备注。
- `job_application_status_log`：投递状态流转日志，记录从哪个状态切到哪个状态、操作者、原因和事件时间。
- `job_application_event`：投递事件表，记录跟进、笔试、面试、反馈、复盘等事件。

其中 `deadline` 是字符串字段，用于兼容不同招聘信息中的截止日期表达；前端“投递时间 / 截止时间 / 下次跟进时间”只要求用户选择日期，不要求录入具体时分秒。

当前前台接口位于：

```text
/api/user/applications/**
```

列表接口支持按 `companyType` 筛选公司类型，并通过 `followUpScope=PENDING|OVERDUE` 查询待跟进或已逾期记录。
