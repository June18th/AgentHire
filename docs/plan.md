## 一、项目概述

### 1.1 项目定位

JobClaw 是一个基于 AI Agent 的校招信息智能管理平台，通过 IM 渠道（微信、钉钉等）与用户交互，提供岗位信息收集、推送、推荐等服务。

### 1.2 V2 版本目标

V2 版本是对 V1 的全面重构，核心目标是：
- **模块化架构**: 将系统拆分为独立的核心模块、通道模块、模型提供者模块
- **多通道支持**: 支持微信、钉钉、飞书等多种 IM 渠道
- **多模型适配**: 支持 OpenAI、智谱、阿里、Anthropic 等多家大模型
- **智能任务管理**: 实现一次性任务、周期性任务、延时任务的完整能力
- **用户隔离**: 按用户 ID 隔离会话、任务、偏好，避免相互干扰

## 三、模块架构

### 3.1 模块划分

```
JobClaw/
├── app/              # 主应用模块（业务逻辑）
├── core/             # 核心模块（框架骨架）
│   ├── channel/      # Channel 抽象层
│   ├── agent/        # Agent 抽象层
│   ├── bus/          # 消息总线
│   ├── tasks/        # 任务管理
│   ├── configuration/# 动态配置
│   ├── memory/       # 记忆管理
│   └── tools/        # 工具集
├── channels/         # IM 通道实现
│   ├── wechat-clawbot/  # 微信 ClawBot
│   └── dingding/        # 钉钉
├── providers/        # 模型提供者
│   ├── openai/       # OpenAI 接口风格
│   ├── zhipu/        # 智谱清言
│   ├── ali/          # 阿里云
│   └── anthropic/    # Anthropic
├── plugins/          # 插件模块
│   └── playwright/   # 网页抓取
└── ui-react/         # 前端应用
```

### 3.2 核心模块职责

#### core/channel
- 定义 Channel 顶层抽象，支持多种渠道扩展
- 实现消息接收、发送的统一接口
- 支持用户绑定/解绑流程

#### core/agent
- 定义 Agent 抽象层
- 实现会话持久化（基于 YAML 文件）
- 记忆管理（工作记忆、短期摘要、长期记忆）

#### core/bus
- 基于发布/订阅模式实现消息转发
- 定义四种消息类型：
  - 接收 IM 消息
  - 推送消息到 IM（问答回复 + 主动推送）
  - 用户绑定消息
  - 用户解绑消息

#### core/tasks
- 实现任务管理能力
- 支持一次性任务、周期性任务
- 任务存储结构：
  - `workspace/tasks/{jobClawUserId}/recurring/` - 周期性任务
  - `workspace/tasks/{jobClawUserId}/年-月-日/时分秒-任务名.md` - 一次性任务

#### core/configuration
- 动态配置管理，支持热更新
- IM 通道动态绑定
- 用户级 API Key 和模型偏好配置

#### core/memory
- 会话持久化（YAML 文件）
- 上下文窗口管理（滑动窗口策略）
- 对话历史压缩（异步摘要生成）
- 长期记忆提取（用户画像）
- 用户身份说明文件管理（agent.md、soul.md、info.md、user.md、tools.md）

## 五、技术挑战与解决方案

### 5.1 工具管理

**问题**：
- 工具太多，全部注册还是按需选择性注册？
- 工具执行失败如何处理（参数缺失、调用失败）？
- 如何实现按需加载工具？
- 敏感工具执行前的授权如何实现？
- 多工具协作场景如何处理（如下单需要多个信息收集工具）？

**思考方向**：
- 分层注册：核心工具常驻，业务工具按需加载
- 添加中间层：工具搜索/发现层，根据上下文动态选择
- 错误处理：重试机制、降级策略、用户提示
- 权限控制：基于角色的工具访问控制
- 工作流编排：使用 LangGraph 实现多步骤工具协作

### 5.2 意图识别

**问题**：
- 为什么要做意图识别？
  - JobClaw 底层提供多个业务 Agent（岗位收集、推送、推荐等）
  - 需要根据用户意图自动路由到合适的 Agent
- 在哪一层实现？
  - IM 消息接收层，通过模型识别意图后路由
- 如何实现？
  - 关键词匹配 / 模型识别 / 预定义激活关键词

**思考方向**：
- 轻量级意图分类器（小模型快速判断）
- 结合规则引擎（关键词 + 正则）
- 维护对话状态机，跟踪当前活跃的 Agent

### 5.3 多轮对话状态一致性

**问题**：
- 用户已与 xxAgent 对话，下一轮如何确保仍与 xxAgent 对话？
- 如果用户突然切换到其他话题，如何正确路由到 yyAgent？
- 如何保障多步骤流程不因中间打断而崩溃？

**思考方向**：
- 会话状态管理：记录当前活跃的 Agent 和上下文
- 显式切换机制：用户可通过命令或关键词切换 Agent
- 隐式检测：模型检测到话题变化时自动切换
- 状态持久化：将对话状态保存到会话文件中
- 回滚机制：支持回到上一个 Agent 或状态

### 5.4 模型可靠性

**问题**：
- 多模型无缝切换如何实现？
- 模型推理和决策的可靠性如何衡量？
- 是无脑接受大模型返回，还是设计衡量标准？

**思考方向**：
- 统一接口抽象：所有 Provider 实现相同接口
- 健康检查：定期检测模型可用性
- 结果验证：对关键操作进行二次确认
- 置信度评分：模型返回时附带置信度
- 人工审核：对低置信度结果进行人工介入

### 5.5 多 Agent 交互协作

**问题**：
- 多 Agent 之间如何进行信息交换？
- 如何相互发现和调用？

**思考方向**：
- Agent 注册中心：维护可用 Agent 列表和能力描述
- 消息总线：通过 Bus 实现 Agent 间通信
- 共享上下文：Agent 可读取共享的会话状态
- 工作流编排：使用 LangGraph 编排多 Agent 协作流程

### 5.6 上下文窗口管理

**问题**：
- 上下文窗口有限（4K/16K/128K），但会话可能无限
- 如何压缩对话而不丢失关键信息？
- 何时触发压缩？丢弃哪些对话？

**思考方向**：
- 分层记忆架构：
  - 工作记忆：最近 N 轮完整保留
  - 短期摘要：较早对话的结构化摘要
  - 长期记忆：向量化的用户偏好和关键事实
- 消息重要性评分：基于规则 + 模型判断
- 多级压缩策略：轻度/中度/重度压缩
- 异步摘要生成：不阻塞用户交互

**详细方案**：参见 [memory/mem.md](core/src/main/java/com/git/hui/jobclaw/core/agent/memory/mem.md)

### 5.7 用户隔离与会话共享

**问题**：
- 不同用户的会话如何隔离？
- 同一用户不同通道的会话如何关联？
- 用户共性特性如何共享？

**思考方向**：
- 按 JobClawUserId 隔离会话文件和任务文件
- 用户画像统一管理：跨通道共享偏好和事实
- 通道映射表：记录用户与多个通道的绑定关系

### 5.8 记忆工程

**问题**：
- 记忆工程有什么用？为什么需要？
- 如何实现？

**思考方向**：
- 短期记忆：会话上下文，保证对话连贯性
- 长期记忆：用户偏好、历史行为、关键事实
- 记忆提取：从对话中自动抽取结构化信息
- 记忆检索：根据上下文按需注入相关记忆
- 记忆更新：增量更新，避免覆盖已有信息

### 5.9 复杂任务拆解与执行

**问题**：
- 复杂任务如何拆解、分布实现？
- 执行过程中出现异常如何处理？
- Plan-and-Execute 的指导原理是什么？

**思考方向**：
- 任务分解：将大任务拆分为子任务 DAG
- 依赖管理：明确子任务间的依赖关系
- 容错机制：单个子任务失败不影响整体
- 状态追踪：实时追踪任务执行进度
- 回滚策略：支持任务级别的撤销和重试

### 5.10 Agent 自主工作

**问题**：
- 如何让 Agent 永不停息地工作，而非一问一答的被动触发？

**思考方向**：
- 任务调度：基于 Cron 的周期性任务
- 事件驱动：监听外部事件（新岗位发布、用户状态变化）
- 主动推送：基于用户偏好主动推送相关信息
- 后台代理：独立的后台 Agent 持续监控和执行任务

### 5.11 自学习与进化

**问题**：
- 如何实现越用越智能、越用越懂你？
- 如何设计拥有自主进化、自学习的 Agent？

**思考方向**：
- 用户画像构建：从对话中提取偏好和事实
- 行为分析：记录用户点击、投递、反馈等行为
- 反馈循环：用户对推荐结果的反馈用于优化模型
- A/B 测试：对比不同策略的效果
- 在线学习：根据新数据持续更新用户画像

### 5.12 安全与监控

**问题**：
- 提示注入攻击如何防范？
- 敏感信息传输是否会导致泄露？
- 越权操作与工具滥用如何控制？
- 合规审计如何做？

**思考方向**：
- 输入隔离：系统指令与用户输入严格分离
- 输出编码：对模型输出进行 sanitization
- 数据脱敏：传输前对敏感信息进行脱敏
- 访问控制：基于角色的工具和 API 访问控制
- 审计日志：记录所有关键操作和模型调用
- 速率限制：防止滥用和 DDoS 攻击

## 七、关键技术决策

### 7.1 为什么选择 YAML 文件存储会话？

**优点**：
- 无需额外数据库依赖，降低部署复杂度
- 人类可读，便于调试和问题排查
- 天然支持版本控制和备份
- 适合小规模应用场景

**缺点**：
- 大规模场景下性能瓶颈
- 并发写入需要文件锁
- 查询能力弱于数据库

**适用场景**：
- 个人或小团队使用
- 会话量不大（< 1000 会话/天）
- 对查询性能要求不高

**未来演进**：
- 当规模扩大时，可迁移到向量数据库或关系型数据库
- 保留 YAML 作为导出/备份格式

### 7.2 为什么采用发布/订阅模式实现消息转发？

**优点**：
- 解耦消息接收方和处理方
- 支持多个消费者同时处理消息
- 易于扩展新的消息类型和处理逻辑
- 支持异步处理和背压控制

**实现方式**：
- 使用 Spring ApplicationEventPublisher
- 定义四种事件类型：ImMessageReceivedEvent、ImMessagePushEvent、UserBindEvent、UserUnbindEvent
- 各模块订阅感兴趣的事件

### 7.3 为什么任务存储使用文件系统而非数据库？

**优点**：
- 与现有会话存储方案保持一致
- 便于手动编辑和调试
- 天然支持版本控制（Git）
- 降低系统复杂度

**缺点**：
- 查询能力弱（无法高效按条件筛选）
- 并发控制复杂
- 事务支持弱

**未来演进**：
- 如需复杂查询，可引入轻量级数据库（SQLite/H2）
- 或使用文件系统 + 索引文件的方式
