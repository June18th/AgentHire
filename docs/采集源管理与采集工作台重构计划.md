# 采集源管理与采集工作台重构计划

> **文档说明**: 本文档整理 2026-06-27 围绕 `求职派Agent`、`录入数据`、`任务队列`、`草稿列表`、`职位列表` 的产品与技术重构讨论，作为后续动工计划。
>
> **核心结论**: 投料是动作，任务是执行记录，采集源是长期资产；Admin 半自动采集和 Agent 自动作业应该使用同一套业务对象，但提供不同工作视角。

---

## 一、问题背景

当前后台菜单和数据流存在几个混在一起的问题：

| 问题 | 当前表现 | 影响 |
| --- | --- | --- |
| 菜单按实现拆分 | `求职派Agent`、`录入数据`、`草稿列表`、`职位列表` 平铺在左侧 | 用户不容易理解“从投料到职位入库”的完整链路 |
| 任务承担过多语义 | `gather_task` 同时表示投料内容、执行尝试、处理结果 | 任务队列中出现很多相似行，难以判断是否真的有新结果 |
| 缺少采集源层 | 同一个链接反复投料会生成多条相似任务 | 系统不知道这是同一个来源的第 N 次执行 |
| 结果分类粗 | 当前结果主要是 `insertDraftIds` 和 `updateDraftIds` | 完全重复的结果也可能显示成“更新草稿 20”，造成误解 |
| Agent 与 Admin 视角混杂 | Agent 作业链和人工投料审核共用底层数据，但入口割裂 | 自动化流程和半自动流程的职责边界不清 |

## 二、核心概念

| 概念 | 定义 | 是否可变 | 示例 |
| --- | --- | --- | --- |
| 采集源 | 长期存在、可反复执行、可追踪去重的数据来源资产 | 可更新，建议保留版本 | 招聘链接、HTML、Excel、图片、官网招聘页、社群来源 |
| 投料 | 用户或 Agent 发起的一次输入动作 | 未执行前可调整，执行后不可改历史 | 粘贴链接后点击提交 |
| 任务 | 对某个投料或采集源执行的一次运行记录 | 不应修改历史，只能重跑生成新记录或重置状态 | 第 3 次执行某招聘链接 |
| 草稿 | 任务产出的待审核职位数据 | 可人工编辑和发布 | AI 抽取出的 20 条岗位草稿 |
| 职位 | 草稿审核发布后的正式职位数据 | 可维护 | 最终展示给用户的岗位 |
| Agent 作业链 | Agent 自动完成投料、采集、清洗、发布的执行流程 | 记录过程，不替代业务对象 | `TaskClassify -> TaskGather -> DraftWasher -> DraftPublish` |

推荐的数据生命周期：

```text
采集源
  -> 投料
  -> 任务
  -> 草稿
  -> 职位
```

Admin 半自动采集和 Agent 自动作业都进入这条链路，但在页面上提供不同视角：

```text
Admin 视角：我投了什么、跑成什么、哪些草稿要审核、哪些职位已入库
Agent 视角：Agent 为什么这么跑、每一步产出什么、失败在哪里、重复率如何
```

## 四、采集源管理设计

### 4.1 采集源管理什么

采集源管理的对象不是某一次任务，而是“可以被反复采集的来源资产”：

| 管理项 | 说明 |
| --- | --- |
| 来源类型 | 招聘链接、文本/HTML、表格/图片、官网、社群、飞书表格等 |
| 来源内容 | URL、文本摘要、附件路径、页面快照等 |
| 来源指纹 | 基于规范化内容生成 `sourceHash`，用于识别同一来源 |
| 来源版本 | 来源内容变化时生成新版本，不覆盖历史任务 |
| 归属视角 | `admin` 主动投料、`agent` 自动发现、定时任务等 |
| 执行模式 | 只生成草稿、Agent 自动清洗、Agent 自动发布 |
| 最近任务 | 最近一次执行任务 ID、状态、耗时、时间 |
| 最近结果 | 新增、更新、无变化、失败数量 |
| 健康状态 | 正常、失效、暂停、归档 |
| 去重策略 | 按来源链接、正文 hash、公司+岗位+地点+更新时间等组合 |

### 4.2 采集源不做什么

| 不建议做法 | 原因 |
| --- | --- |
| 直接修改已完成任务的投料内容 | 任务是历史记录，修改后无法追溯当时实际执行内容 |
| 把采集源等同于任务 | 会继续产生重复任务堆叠的问题 |
| 把采集源做成纯 UI 分组 | 如果没有 sourceId/hash/resultDiff，重复判断仍然解决不了 |

推荐规则：

| 场景 | 处理方式 |
| --- | --- |
| 任务未开始 | 可以编辑或取消 |
| 任务执行中 | 不允许编辑，只能停止或等待 |
| 任务已完成 | 不改历史，基于同一采集源重新运行 |
| 来源内容变化 | 生成新版本，再创建新任务 |
| 来源没变且结果没变 | 任务结果显示“无变化”，不再显示“更新草稿 20” |

## 五、实施计划表

### 阶段 1：信息架构与页面骨架

| 任务 | 目标 | 交付物 | 验收标准 |
| --- | --- | --- | --- |
| 重构后台导航 | 从左侧平铺菜单改为“顶部工作域 + 工作域内左侧菜单” | 新后台导航结构 | 配置项不再和业务流程平铺；采集链路集中在采集工作台 |
| 重命名页面语义 | 将 `录入数据` 改为 `采集工作台`，`求职派Agent` 改为 `Agent作业台` | 菜单文案和页面标题更新 | 用户能按“投料 -> 任务 -> 草稿 -> 职位”理解流程 |
| 采集工作台整合 | 将投料、任务、草稿、职位作为同一工作域内入口 | 页面入口和跳转关系 | 任务结果能跳转草稿，草稿能发布到职位 |
| Agent 作业台定位 | 保留 Agent 自动链路观察能力 | `/admin/progress` 语义升级 | 页面强调作业链、执行记录、Agent 产出 |

### 阶段 2：采集源数据模型

| 任务 | 目标 | 交付物 | 验收标准 |
| --- | --- | --- | --- |
| 新增采集源表 | 把长期来源从任务中拆出来 | `gather_source` 或同等实体 | 同一链接/内容可归入同一个 sourceId |
| 任务关联采集源 | 任务成为某个来源的一次执行记录 | `gather_task.source_id`、`source_version`、`runner_type` | 任务列表能按来源和执行方式过滤 |
| 来源指纹 | 判断投料是否重复 | `source_hash`、`content_hash` | 相同规范化内容不会创建重复来源 |
| 来源版本 | 保留来源变更历史 | 来源版本字段或版本表 | 重新采集时可追溯每次使用的来源内容 |

### 阶段 3：结果差异与去重升级

| 任务 | 目标 | 交付物 | 验收标准 |
| --- | --- | --- | --- |
| 结果分类扩展 | 区分新增、更新、无变化、跳过、失败 | 任务结果结构扩展 | 重复结果显示“无变化 20”，不再误显示“更新草稿 20” |
| 草稿差异判断 | 判断已有草稿是否真的变化 | 草稿内容 hash 或字段级比较 | 字段无变化时不刷新更新时间和未处理状态 |
| 来源级汇总 | 在采集源列表展示最近执行结果 | 来源汇总字段或查询聚合 | 来源列表能看到最近新增/更新/无变化数量 |
| 重跑策略 | 明确重跑是复用来源还是复制任务 | `重新采集` 操作 | 重跑不会破坏历史任务内容 |

### 阶段 4：采集源管理页面

| 页面 | 关键能力 | 验收标准 |
| --- | --- | --- |
| 来源列表 | 搜索、类型筛选、状态筛选、最近任务、最近结果 | 能快速识别哪些来源有效、重复、失效 |
| 来源详情 | 原始内容、版本、任务历史、草稿结果 | 能从来源追溯到每次任务和产出草稿 |
| 来源操作 | 重新采集、暂停、归档、查看草稿、查看任务 | 常用操作不需要回到任务队列里找记录 |
| 来源健康 | 失败次数、最后成功时间、重复率 | 能判断是否应该暂停或调整来源 |

### 阶段 5：Agent 与 Admin 视角统一

| 任务 | 目标 | 交付物 | 验收标准 |
| --- | --- | --- | --- |
| 执行方式标识 | 区分 Admin 投料和 Agent 自动发现/执行 | `runner_type` 或同等字段 | 任务、草稿、职位都能按来源视角过滤 |
| Agent 结果回链 | Agent 作业链产出回写采集源、任务、草稿 | 作业链与采集源关联 | Agent 页面能看到对应来源和草稿 |
| 复用列表页面 | 避免复制四套页面 | 列表支持过滤参数 | `Agent任务/草稿/职位` 是同一页面的过滤视角 |
| 自动化边界 | Agent 可自动清洗/发布，但保留审核模式 | 执行模式配置 | 不同来源可设置不同自动化程度 |

## 六、建议接口与数据约定

### 6.1 采集源基础字段

| 字段 | 说明 |
| --- | --- |
| `id` | 采集源 ID |
| `type` | 来源类型 |
| `title` | 来源标题或摘要 |
| `content` | 来源内容或文件路径 |
| `sourceHash` | 规范化来源指纹 |
| `contentHash` | 当前版本内容指纹 |
| `version` | 来源版本 |
| `ownerType` | `admin` / `agent` / `system` |
| `runnerMode` | `draft_only` / `agent_wash` / `agent_publish` |
| `status` | `active` / `paused` / `archived` / `invalid` |
| `lastTaskId` | 最近任务 ID |
| `lastResultSummary` | 最近结果摘要 |
| `createTime` / `updateTime` | 时间字段 |

### 6.2 任务结果结构

建议把当前任务结果从“新增/更新”扩展为：

```json
{
  "msg": "success",
  "insertDraftIds": [1, 2],
  "updateDraftIds": [3, 4],
  "unchangedDraftIds": [5, 6],
  "skipDraftIds": [],
  "failedItems": [],
  "summary": {
    "insert": 2,
    "update": 2,
    "unchanged": 2,
    "skip": 0,
    "failed": 0
  }
}
```

## 七、优先级建议

| 优先级 | 内容 | 原因 |
| --- | --- | --- |
| P0 | 导航和命名重构 | 先解决用户理解成本，让后台结构清晰 |
| P0 | 任务结果“无变化”分类 | 直接解决任务队列相似行难理解的问题 |
| P1 | 采集源实体和 sourceId 关联 | 从根上解决同一来源重复任务无法聚合的问题 |
| P1 | 采集源列表/详情 | 让来源成为可管理资产 |
| P2 | 来源版本和健康状态 | 为长期运营、定时采集、Agent 自动发现做准备 |
| P2 | Agent/Admin 过滤视角统一 | 降低页面复制和后续维护成本 |

## 八、分轮动工计划

### 8.1 第一轮：导航重构与采集链路收口

第一轮目标是先解决“用户看不懂后台结构”的问题，不一次性动完整后端模型。

| 任务 | 说明 |
| --- | --- |
| 重构后台导航 | 改为“顶部工作域 + 工作域内左侧菜单” |
| 建立工作域 | 顶部至少包含 `采集工作台`、`Agent作业台`、`采集源管理`、`平台配置` |
| 重命名现有页面 | `录入数据` 改为 `采集工作台`，`求职派Agent` 改为 `Agent作业台` |
| 收口采集链路 | 在采集工作台内放 `新建投料`、`采集任务`、`草稿审核`、`职位库` |
| 建立采集源入口 | 新增 `采集源管理` 菜单和空态/轻量页面，先承载概念和后续入口 |
| 优化任务结果展示 | UI 层先表达 `新增 / 更新 / 无变化`，为后端结果结构升级预留位置 |

第一轮验收目标：

| 验收项 | 标准 |
| --- | --- |
| 用户能看懂后台结构 | 进入后台后能明确区分采集工作台、Agent 作业台、采集源管理、平台配置 |
| 采集链路清楚 | 能按投料、任务、草稿、职位完成一次半自动流程 |
| Agent 视角独立 | Agent 页面不再像普通录入页面，而是作业链观察台 |
| 后续演进有承载点 | 采集源管理入口和术语已建立，可继续扩展 sourceId/hash/version |

### 8.2 第二轮：采集源最小闭环

第二轮目标是让“采集源”从菜单概念变成真实数据对象，解决同一个来源反复生成相似任务的问题。

| 任务 | 说明 |
| --- | --- |
| 新增采集源实体 | 增加 `gather_source` 或等价实体，记录来源类型、内容摘要、hash、状态、最近任务 |
| 投料自动归源 | `asyncSubmit` 和 `agentSubmit` 提交时先计算来源指纹，相同来源归到同一个 sourceId |
| 任务关联来源 | `gather_task` 增加 `source_id`、`source_version`、`runner_type` 等关联字段 |
| 来源列表页面 | 展示来源类型、来源摘要、归属、状态、最近任务、最近结果、更新时间 |
| 来源详情页面 | 展示原始内容、版本信息、任务历史、关联草稿入口 |
| 重新采集动作 | 从来源发起新任务，不修改历史任务投料 |

第二轮验收目标：

| 验收项 | 标准 |
| --- | --- |
| 同源任务可聚合 | 同一个招聘链接或相同内容再次投料时，可以归到同一个采集源 |
| 任务历史可追溯 | 从采集源详情能看到每一次任务执行记录 |
| 来源能直接操作 | 可以从来源列表或详情发起重新采集、查看任务、查看草稿 |
| 历史任务不被污染 | 已完成任务的 content 不因来源编辑而变化 |

### 8.3 第三轮：结果差异、去重和 Agent/Admin 视角统一

第三轮目标是解决“重复任务看起来像更新成功”的业务误导，并把 Agent 自动链路和 Admin 半自动链路统一到同一套对象上。

| 任务 | 说明 |
| --- | --- |
| 扩展任务结果结构 | 在 `insertDraftIds`、`updateDraftIds` 外增加 `unchangedDraftIds`、`skipDraftIds`、`failedItems` 和 summary |
| 草稿内容差异判断 | 对已有草稿做字段级比较或内容 hash，无变化时不刷新更新时间和未处理状态 |
| 任务列表结果升级 | 任务结果明确显示新增、更新、无变化、失败，不再把重复结果都归为更新 |
| 来源级统计 | 采集源列表展示最近新增、更新、无变化、失败数量和重复率 |
| Agent/Admin 过滤统一 | 任务、草稿、职位列表支持按 `runner_type`、`source_id`、`ownerType` 过滤 |
| Agent 作业回链 | Agent 作业链中的任务、草稿、职位都能回到对应采集源 |

第三轮验收目标：

| 验收项 | 标准 |
| --- | --- |
| 重复结果可识别 | 同源同内容重复执行时，任务显示“无变化 N”，而不是“更新草稿 N” |
| 草稿状态不被误刷新 | 字段没有变化的草稿不重新进入待处理状态 |
| Agent 与 Admin 共用底层对象 | 同一批来源、任务、草稿、职位能按不同视角过滤查看 |
| 来源运营有指标 | 能看到来源重复率、最近成功时间、连续失败次数等基础健康信息 |

### 8.4 后续增强：来源运营与自动化策略

前三轮完成后，再考虑更长期的运营能力：

| 能力 | 说明 |
| --- | --- |
| 来源版本管理 | 来源内容变化时保存版本，支持对比不同版本任务结果 |
| 定时采集 | 对稳定来源配置周期性采集 |
| 来源健康告警 | 连续失败、内容为空、重复率过高时提示暂停或检查 |
| 自动化策略 | 不同来源配置 `只生成草稿`、`自动清洗`、`自动发布` |
| Agent 自动发现来源 | Agent 从页面或社群信息中发现新来源后进入采集源池 |
