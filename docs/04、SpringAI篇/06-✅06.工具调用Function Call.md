# 06-✅06.工具调用Function Call

我们知道大模型是基于一系列数据进行训练的，且每次训练的成本很高；那大模型是怎么样表现得十八般武艺样样精通呢？


Function Call 就是设计出来给大模型当外挂的瑞士军刀，通过给大模型一个与外部工具进行交互的方式，来扩展大模型在训练时缺失的能力。

### 一、工作原理
SpringAi 提供了完备的工具调用封装，可以非常方便的将本地方法封装成工具，供大模型调用。

#### 1. 工具调用流程

![](https://github.com/liuyueyi/spring-ai-demo/raw/master/docs/static/06-1.webp)

Function Call 这事儿，其实可以想象成一次“接力赛”。


一开始，我们在向大模型发请求的时候，会把工具的基本信息一并塞过去：工具叫什么、能干嘛、要啥参数，这些都告诉它。接着，大模型就像个指挥官，它会根据这些信息，决定该用哪个工具，并生成一条带着工具名字和参数的调用请求。


应用程序拿到这条指令，就知道该找谁上场了，于是点名对应的工具去执行。工具干完活，把结果交回给应用程序，再由应用程序转手交还给大模型。


最后，大模型把工具提供的结果融进它的整体回答里，打磨成一份给用户的最终回复。

#### 2. SpringAI关键实现
SpringAI提供了两种内置方式将本地方法封装成工具

+ 声明式：通过注解 `@Tool`
+ 编程式：通过底层的 `MethodToolCallback` `FunctionToolCallback`


![](https://github.com/liuyueyi/spring-ai-demo/raw/master/docs/static/06-2.webp)

上图中橙色的部分，就是SpringAI将本地方法封装成工具，供大模型进行交互；接下来我们看一下SpringAI的关键类。

##### a. ToolCallback
ToolCallback 接口定义了 AI 模型可调用工具的方式，包含工具定义和执行逻辑。

```java
public interface ToolCallback {
    ToolDefinition getToolDefinition();

    default ToolMetadata getToolMetadata() {
        return ToolMetadata.builder().build();
    }

    String call(String toolInput);

    default String call(String toolInput, @Nullable ToolContext tooContext) {
        if (tooContext != null && !tooContext.getContext().isEmpty()) {
            throw new UnsupportedOperationException("Tool context is not supported!");
        } else {
            return this.call(toolInput);
        }
    }
}
```

SpringAI提供了两种内置实现

+ MethodToolCallback: 将本地方法定义为AI模型可调用的工具
+ FunctionToolCallback: 将函数定义为AI模型可调用的工具

##### b. ToolDefinition
ToolDefinition 接口提供了 AI 模型识别工具可用性所需的信息，包括工具名称、描述及输入模式。每个 ToolCallback 实现必须提供 ToolDefinition 实例来定义工具。

```java
public interface ToolDefinition {

    /**
	 * 工具名
	 */
    String name();

    /**
	 * 工具说明，告诉AI这个工具可以干嘛
	 */
    String description();

    /**
	 * 传参方式，通常是 json schema格式
	 */
    String inputSchema();
}
```

如果需要手动创建工具，可以使用`ToolDefinition.Builder`

```java
ToolDefinition toolDefinition = ToolDefinition.builder()
    .name("currentWeather")
    .description("Get the weather in location")
    .inputSchema("""
        {
            "type": "object",
            "properties": {
                "location": {
                    "type": "string"
                },
                "unit": {
                    "type": "string",
                    "enum": ["C", "F"]
                }
            },
            "required": ["location", "unit"]
        }
    """)
    .build();
```

除了上面的方式，对于本地的 java方法，可以直接通过反射的方式来生成基于方法的工具定义

```java
Method method = ReflectionUtils.findMethod(DateTimeTools.class, "getCurrentDateTime");
ToolDefinition toolDefinition = ToolDefinition.from(method);

// 如果需要显示修改某些属性，可以如下
ToolDefinition toolDefinition = ToolDefinition.builder(method)
     .name("currentDateTime")
     .description("Get the current date and time in the user's timezone")
     .inputSchema(JsonSchemaGenerator.generateForMethodInput(method))
     .build();
```

##### c. JSON Schema
在给大模型提供工具时，需要先把“工具的长相”告诉它——也就是 function call 的输入类型模式。大模型只有知道参数该怎么传，才会乖乖地按规矩来调用。


Spring AI 在这方面很贴心，它内置了一个 JsonSchemaGenerator 类，可以直接帮我们把工具的输入类型转成 JSON Schema。换句话说，不用我们自己手搓 Schema，Spring AI 就能自动生成，省心不少。


下面是一个json schema 示例（在前面[结构化返回](https://github.com/liuyueyi/spring-ai-demo/blob/master/docs/03.%E7%BB%93%E6%9E%84%E5%8C%96%E8%BF%94%E5%9B%9E.md)中，实际上也用到了 JSON Schema，用于告知大模型如何返回我们需要的数据类型）

```json
{
  "type": "object",
  "properties": {
    "location": {
      "type": "string"
    },
    "unit": {
      "type": "string",
      "enum": ["C", "F"]
    }
  },
  "required": ["location", "unit"]
}
```

这里的json schema主要用于方法/函数调用的传参定义，SpringAI支持下面四种基于注解生成的输入参数描述、是否必传

+ `Spring AI` 的 `@ToolParam(description = "…", required=false)`： 参数描述 + 是否必传
+ `Jackson` 的 `@JsonClassDescription(description = "…)`: 参数描述
+ `Jackson` 的 `@JsonPropertyDescription(description = "…")` : 参数描述
+ `Jackson` 的 `@JsonProperty(required = false)` : 参数是否必传
+ `Swagger` 的` @Schema(description = "…", required = false)` ： 参数描述 + 是否必传
+ `Spring` 的 `@Nullable` ： 参数是否必传

示例：

```java
class DateTimeTools {
    @Tool(description = "Set a user alarm for the given time")
    void setAlarm(@ToolParam(description = "Time in ISO-8601 format", required = true) String time) {
        LocalDateTime alarmTime = LocalDateTime.parse(time, DateTimeFormatter.ISO_DATE_TIME);
        System.out.println("Alarm set for " + alarmTime);
    }
}
```

##### d. 结果转换
通常一个方法/函数返回的都是 Object 对象，但大模型没法直接用。为了把结果喂给大模型，Spring AI 定义了一个 ToolCallResultConverter，专门负责把返回值序列化成 String。


默认情况下，Spring AI 用的是基于 Jackson 的 JSON 序列化实现，也就是 DefaultToolCallResultConverter。大多数场景下都能开箱即用，直接就能把返回结果转成 JSON 串交给模型。


当然，如果你有自己的偏好，比如想用 Gson、Fastjson，或者对序列化格式有特殊要求，也可以自定义一个实现来替换掉默认的。

```java
@FunctionalInterface
public interface ToolCallResultConverter {

    /**
	 * Given an Object returned by a tool, convert it to a String compatible with the
	 * given class type.
	 */
    String convert(@Nullable Object result, @Nullable Type returnType);

}
```


##### e. ToolContext
Spring AI 还支持通过 **ToolContext API** 给工具额外传递上下文信息。简单来说，就是在你调用 ChatClient 的时候，可以顺手把一些额外的数据带过去，这些数据会被装进 ToolContext 里，然后在工具调用的时候一并传入。


ToolContext 的实现其实非常简单，本质上就是一个不可变的 Map，用来承接这些上下文数据。这样一来，工具调用的时候就能拿到更多环境信息，不至于只依赖函数参数。


后面我会用一个具体的示例，来展示 ToolContext 是怎么派上用场的。

```java
public final class ToolContext {
    public static final String TOOL_CALL_HISTORY = "TOOL_CALL_HISTORY";
    private final Map<String, Object> context;

    public ToolContext(Map<String, Object> context) {
        this.context = Collections.unmodifiableMap(context);
    }

    public Map<String, Object> getContext() {
        return this.context;
    }

    public List<Message> getToolCallHistory() {
        return (List)this.context.get("TOOL_CALL_HISTORY");
    }
}
```

#### 2. 声明式使用方式
我们来定义一个最简单的工具方法：获取当前时间。实现起来很轻松，只要在方法上加上 @Tool 注解就行。


这里要特别提醒一点：**description 属性非常关键**。它的作用是告诉大模型——什么场景下该用这个工具。如果描述得模糊不清，大模型可能在该调用的时候不调用，甚至还会误用别的工具。


所以，写工具的时候，description 不是可有可无的“注释”，而是真正影响模型调用行为的“说明书”。

```java
class DateTimeTools {

    // 获取当前时间，工具描述，使用中文也是可以的
    @Tool(description = "Get the current date and time in the user's timezone")
    String getCurrentDateTime() {
        String ans = LocalDateTime.now().atZone(LocaleContextHolder.getTimeZone().toZoneId()).toString();
        System.out.println("进入获取当前时间了：" + ans);
        return ans;
    }
}
```

接下来，我们使用 `ChatClient` 来调用工具。

```java
@RestController
public class ChatController {

    private final ChatClient chatClient;

    public ChatController(ZhiPuAiChatModel chatModel) {
        this.chatClient = ChatClient.builder(chatModel)
        .build();
    }

    @RequestMapping(path = "time")
    public String getTime(String msg) {
        return chatClient.prompt(msg).tools(new DateTimeTools()).call().content();
    }

    @RequestMapping(path = "timeNoTools")
    public String getTimeNoTools(String msg) {
        return chatClient.prompt(msg).call().content();
    }
}
```

上面有提供了两个接口， getTime 方法使用了工具，getTimeNoTools 方法没有使用工具；使用方式形如`chatClient.prompt().tools(xxx)`：通过tools方法注入工具。


接下来我们看一下实际的体验。


![](https://github.com/liuyueyi/spring-ai-demo/raw/master/docs/static/06-3.webp)

说明：在请求 `time` 方法时，正确的返回了时间，但是大模型并不总是和我们预期一致，同样的提问，它有时不会主动调用注入的工具。

#### 3. 带参数的工具方法
前面我们写了一个“无参数”的小工具，现在来看看**带参数的工具方法**。


在方法参数上，可以用 @ToolParam 注解来解释说明（当然，这个注解不是必填的）。它有两个常用属性：

+ **description**：参数描述，用来帮助模型更准确地理解该怎么用这个参数。比如可以写清楚格式要求、允许取值范围等。
+ **required**：指定这个参数是不是必填项。默认值是 true，也就是说没特别声明的话，所有参数都是必需的。


举个例子，我在 DateTimeTools 里定义了一个方法，用来返回不同时区的当前时间，并且供大模型调用。参数就是“时区”，通过 @ToolParam 写上解释说明，大模型就能更清楚什么时候、该怎么传这个参数。这样一来，上面提到的 API 就能直接用了。

```java
class DateTimeTools {
    @Tool(description = "传入时区，返回对应时区的当前时间给用户")
    String getTimeByZoneId(@ToolParam(description = "需要查询时间的时区") ZoneId area) {
        // 根据时区，查询对应的时间
        ZonedDateTime time = LocalDateTime.now().atZone(area);

        // 转换为 2025-07-26 20:00:00 格式的字符串
        // 将输入时区的时间转换为本地时区
        ZonedDateTime localTime = time.withZoneSameInstant(ZoneId.systemDefault());
        DateTimeFormatter formatter = DateTimeFormatter.ofPattern("yyyy-MM-dd HH:mm:ss");
        String ans = localTime.format(formatter);
        System.out.println("传入的时区是：" + area + "-" + ans);
        return ans;
    }
}
```


![](https://github.com/liuyueyi/spring-ai-demo/raw/master/docs/static/06-4.webp)

从上面的访问示例也可以看出，我们询问某个地区的时间时，大模型会自动根据地区找到时区，然后调用我们的工具返回结果。


但是一个问题来了，直接问时间时，它不会调用我们之前定义的那个方法，返回时间，这是为什么呢？


我们尝试调整一下获取当前时间的工具方法说明

```java
@Tool(description = "不需要关注用户时区，直接返回当前的时间给用户")
String getCurrentDateTime() {
    String ans = LocalDateTime.now().atZone(LocaleContextHolder.getTimeZone().toZoneId()).toString();
    System.out.println("进入获取当前时间了：" + ans);
    return ans;
}
```

然后再多问几次，你就会发现大模型会主动调用我们定义的这个方法，返回时间（但是并不能始终保证它总是表现正确），所以一个精确的工具描述，对于大模型的决策非常重要。

#### 4. 添加默认工具
上面说的是用户在发起对话时，可以通过设置 Prompt 来指定工具调用。其实，Spring AI 还提供了另一种更省心的方式——**默认工具设置**。


在创建 ChatClient 的时候，你就能直接指定一组默认工具。这样一来，只要用这个 ChatClient 进行对话，这些工具都会自动提供给大模型使用，不需要每次都手动配置。


用法也很直观，大概长这样：

```java
ChatClient chatClient = ChatClient.builder(chatModel)
    .defaultTools(new DateTimeTools())
    .build();
```

#### 5. ChatModel 添加工具
前面我们一直是基于 ChatClient 来添加工具。那如果在某些场景下，我们直接用 ChatModel 来对话呢？同样也可以加上工具，只不过方式稍微不一样，需要借助 **ChatOptions**。


具体来说，可以通过 ToolCallbacks 来获取一组工具，然后用 ToolCallingChatOptions 把这组工具交给模型。写法大概是这样的：

```java
@RequestMapping(path = "showTime")
 public String showTime(String msg) {
     ToolCallback[] tools = ToolCallbacks.from(new DateTimeTools());
     ChatOptions options = ToolCallingChatOptions.builder()
             .toolCallbacks(tools)
             .build();
     return chatModel.call(new Prompt(msg, options)).getResult().getOutput().getText();
 }
```

#### 6. 编程式使用方式
前面介绍的都是基于 @Tool 注解的声明式工具定义，这种方式很适合工具完全由我们自己实现、可控的场景。


但如果工具实现不是我们写的（比如三方 SDK），或者工具参数需要动态生成，那就要换一种思路——用**编程式定义工具**。


编程式的写法主要是借助 MethodToolCallback.Builder 来构建我们需要的 MethodToolCallback。用之前最好先回顾一下开头第一节里提到的 ToolCallback 和 ToolDefinition 的类结构，有了这个基础，后面看参数设置就会轻松很多（也更容易理解为什么要这样设计）。


几个关键点：

+ **toolDefinition**：定义工具的“身份证”，包括名称、描述和输入模式，是必填项。可以通过 ToolDefinition.Builder 来构建。
+ **toolMetadata**：一些额外的配置，不是必需的。通过 ToolMetadata.Builder 构建。比如其中的 returnDirect 参数，如果设为 true，工具执行结果会直接返回给用户，而不是先给大模型。
+ **toolMethod**：工具方法的 Method 实例（必需）。
+ **toolObject**：包含工具方法的对象实例。如果方法是静态的，那就可以省略。
+ **toolCallResultConverter**：用于把工具调用结果转换成字符串再返回给大模型。如果不配置，会默认使用 DefaultToolCallResultConverter。

```java
@RequestMapping(path = "timeByCodeTool")
public String getTimeByCodeTool(String msg) {
        Method method = ReflectionUtils.findMethod(DateTimeTools.class, "getTimeByZoneId", ZoneId.class);

        ToolDefinition toolDefinition = ToolDefinition.builder()
        .name("getTimeByZoneId")
        .description("传入时区，返回对应时区的当前时间给用户")
        .inputSchema(JsonSchemaGenerator.generateForMethodInput(method))
        .build();

        ToolMetadata toolMetadata = ToolMetadata.builder()
        .returnDirect(false)
        .build();

        ToolCallback callBack = MethodToolCallback.builder()
        .toolDefinition(toolDefinition)
        .toolMetadata(toolMetadata)
        .toolMethod(method)
        .toolObject(new DateTimeTools())
        .build();
        return chatClient.prompt(msg).toolCallbacks(callBack).call().content();
}
```

在使用方法工具时，需要注意一些限制。以下类型目前不支持作为工具方法的参数或返回类型：

+ Optional
+ 异步类型（如 CompletableFuture、Future）
+ 响应式类型 （如 Flow、Mono、Flux）
+ 函数式类型（如 Function、Supplier、Consumer）

#### 7. 函数作为工具调用的使用示例
到现在为止，我们都是将方法作为工具给大模型调用，在java8+之后，除了方法还有些函数式接口，比如`Function`、`Consumer`、`Supplier`、`Predicate`等，这些函数式接口也可以作为工具进行调用，使用方式如下。


通过 `FunctionToolcallback` 来将函数式类型转换为工具。

```java
class NowService implements Function<AreaReq, AreaResp> {
    @Override
    public AreaResp apply(AreaReq req) {
        ZoneId area = req.zoneId();
        // 根据时区，查询对应的时间
        ZonedDateTime time = LocalDateTime.now().atZone(area);

        // 转换为 2025-07-26 20:00:00 格式的字符串
        // 将输入时区的时间转换为本地时区
        ZonedDateTime localTime = time.withZoneSameInstant(ZoneId.systemDefault());
        DateTimeFormatter formatter = DateTimeFormatter.ofPattern("yyyy-MM-dd HH:mm:ss");
        String ans = localTime.format(formatter);
        System.out.println("传入的时区是：" + area + "-" + ans);
        return new AreaResp(ans);
    }
}

record AreaReq(@ToolParam(description = "需要查询时间的时区") ZoneId zoneId) {
}

record AreaResp(String time) {
}

@RequestMapping(path = "timeByCodeFunc")
public String getTimeByCodeFunc(String msg) {
    // 使用函数式工具需要注意的是，传参和返回结果，要么是void，要么是POJO
    ToolCallback callBack = FunctionToolCallback.builder("nowDateByArea", new NowService())
            .description("传入时区，返回对应时区的当前时间给用户")
            .inputType(AreaReq.class)
            // 下面这一行实际是可以省略的，默认就是根据 inputType 进行生成jsonSchema
            .inputSchema(JsonSchemaGenerator.generateForType(AreaReq.class))
            .toolMetadata(ToolMetadata.builder().returnDirect(false).build())
            .build();
    return chatClient.prompt(msg).toolCallbacks(callBack).call().content();
}
```

具体的使用方式，和上面介绍的方法的编程式相差不大，但是有一个需要格外注意的事。


函数输入和输出可以是 void 或 POJO。输入和输出的 POJO 必须是可序列化的，因为需要将结果序列化并发送回模型。函数及输入输出类型必须是 public 的。


以下类型目前不支持作为工具函数的输入或输出类型：

+ 基本类型
+ Optional
+ 集合类型 （如 List、Map、Array、Set）
+ 异步类型（如 CompletableFuture、Future）
+ 响应式类型（如 Flow、Mono、Flux）

#### 8. 动态范式 @Bean
Spring AI 也支持一种类似“声明式”的函数工具注册。原理是通过 ToolCallbackResolver 接口来解析，具体的实现类是 SpringBeanToolCallbackResolver。


在这个模式下，可以直接把任意 Function、Supplier、Consumer 或 BiFunction 类型的 Spring Bean，当成工具使用：

+ **Bean 名称**会自动作为工具名称。
+ **工具描述**则通过 Spring Framework 的 @Description 注解提供。


理论上，写个 @Bean，标上 @Description，就能直接变成工具。但我自己实测下来，并没有跑通，还需要再确认这个逻辑是不是和官网文档保持一致。

### 三、小结
本篇主要讲了 **Spring AI 如何定义和使用工具调用**。大体分为两类：**方法工具 / 函数工具。**

+ 声明式：通过 @Tool 注解直接定义工具，简单直观。
+ 编程式：借助 MethodToolCallback 或 FunctionToolCallback 构建，更灵活，适合三方 SDK 或动态参数的场景。


在工具的定义过程中，需要特别注意三个点：**工具描述、参数、返回值**。其中参数是以 **JSON Schema** 的形式交给大模型的，这一步很关键，描述不清楚可能直接导致模型调用错误。


在默认行为下，Spring AI 会自动帮我们把这一整套流程跑通：拦截模型发出的工具调用请求 → 调用工具 → 拿到结果 → 返回给模型。对开发者来说，这些都是透明的，由各个 ChatModel 内部通过 ToolCallingManager 实现。


![](https://github.com/liuyueyi/spring-ai-demo/raw/master/docs/static/06-6.webp)


当我们需要给模型提供工具时，最开始会把工具的定义一并放进聊天请求（Prompt）里，然后通过 ChatModel API 发给 AI 模型。


如果模型决定调用工具，它会在响应（ChatResponse）里返回工具名称，以及一组符合定义模式的输入参数。ChatModel 收到后，会把这个调用请求转发给 ToolCallingManager。


接下来 ToolCallingManager 会负责识别应该调用哪个工具，并带上参数去执行它。工具执行完成后，结果会先返回给 ToolCallingManager，再转交给 ChatModel。随后，ChatModel 会把这个结果打包成 ToolResponseMessage，传回给 AI 模型。AI 模型拿到工具返回值，就能把它作为额外上下文，生成最终的回答；最后再通过 ChatClient 返回给调用方（ChatResponse）。


目前整个工具调用的执行过程，都是由 Spring AI 托管的。也就是说，内部的工具选择和调用逻辑对用户来说是透明的，不需要你手动干预。当然，如果你希望自己掌控工具执行的生命周期，也可以把 ToolCallingChatOptions 里的 internalToolExecutionEnabled 属性设为 false，这样就能完全自己来控制。这部分内容我会在下一章节展开介绍。
