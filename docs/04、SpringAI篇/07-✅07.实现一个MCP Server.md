# 07-✅07.实现一个MCP Server

MCP可以说是25年最火的协议，全称 Model Context Protocol，中文的意思是模型上下文协议，由 Anthropic 公司于 2024 年底开源的标准化通信协议，旨在解决大模型与外部工具、数据源之间的碎片化集成问题，被誉为 “AI 领域的 USB-C 接口”。


其核心作用是通过统一的协议规范，让大模型（如 Claude、GPT-4、LLaMA 等）能够安全、高效地连接任意工具和数据，而无需为每个系统单独开发适配代码。

### 一、MCP简介
#### 1.1 MCP是什么
MCP 提供了一个通用的开放标准，把 AI 系统和各种数据源连接起来，用一套统一协议替代过去那种零散、各自为政的集成方式。这样一来，AI 就能更简单、更可靠地访问需要的数据。换句话说，MCP 是一个把 AI 助手和“数据所在的系统”（比如内容库、业务工具、开发环境）打通的新标准，目的就是让前沿模型能给出更好、更相关的回答。


如果用一句大白话总结：**MCP 就是先定义一个统一的标准接口，所有想给大模型提供能力的人，只要按这个标准把功能封装好，大模型就能直接加载使用。**


就拿大家都熟悉的 USB 来说。如果只买了台主机，除了开机关机，好像什么都干不了。但当你买了键盘、鼠标、显示器、摄像头，把它们通过 USB 插上去，电脑就立马能识别这些外设，就能正常操作了。这背后的关键就是：电脑和外设都遵循了 USB 协议，所以才能用同一个频道对话。


MCP 做的事和 USB 很像，只不过它面对的不是电脑外设，而是大模型。通过 MCP，大模型和那些给它“外挂新能力”的服务就能说同一种语言，从而真正同频交流。

#### 1.2 MCP总体架构
MCP 遵循客户端-服务端（CS）的架构，一个主机可以连接多个 MCP Server。


![](https://github.com/liuyueyi/spring-ai-demo/raw/master/docs/static/07-2.webp)

在 MCP 架构里，有几个核心角色：

+ **MCP Hosts（主机）**：指通过 MCP 协议获取数据的程序，比如 Claude Desktop、IDE 或者各种 AI 工具。它们是最终使用数据的一端。
+ **MCP Clients（客户端）**：作为调用方存在，负责发起请求，通常和 MCP Server 成对出现，用于建立连接，实现双方通信。
+ **MCP Servers（服务端）**：作为提供方，负责实现 MCP 协议并提供具体的功能。


除了这三位核心角色，在 MCP 的架构里，还常常会看到两个重要的“外部资源”：

+ **Local Data Sources（本地数据源）**：指 MCP Server 所在机器上的本地资源，比如文件、数据库、应用服务等。MCP Server 可以通过读、写、访问这些本地资源，来对外暴露相应的能力。
+ **Remote Services（远程服务）**：指 MCP Server 通过调用外部 API 或服务，来获取或处理数据，再把这些能力通过 MCP 协议暴露出来。

#### 1.3 MCP工作原理
MCP 本质上只是一个协议，本身并不直接提供能力或服务，就像 HTTP 一样。我们日常能感受到的，其实是跑在协议之上的各种应用。


拿 HTTP 来类比：我们打开浏览器，点开一个网页，浏览器通过 HTTP 协议去请求后台服务器，服务器再把网页内容返回来，最终渲染成可视化的页面，这样我们才能“看到”信息。真正让我们有感知的，是浏览器和网站，而不是 HTTP 协议本身。


MCP 也是同样的逻辑。协议只是“语言”和“规则”，真正的能力要靠上层应用来实现。那它的工作流程大概是这样：

1. 用户在客户端里发起一个问题
2. 客户端把问题传给 LLM
3. LLM 分析问题，并决定调用哪个工具
4. 通过 MCP 协议，把工具调用指令发到服务端
5. 服务端执行工具，并把结果返回给 LLM
6. LLM 基于工具结果生成最终回答
7. 回答通过客户端展示给用户

#### 1.4 SpringAI MCP 实现
SpringAI提供了MCP的Client/Server的集成，对于上层业务而言，要想实现一个MCP Server/Client 非常简单。

##### **MCP客户端**
在 MCP 架构里，**MCP 客户端**是核心角色之一。它的职责就是建立并维护与 MCP 服务器的连接，相当于“对话窗口”的一端，负责把大模型的请求和服务器的能力打通。


它的主要工作可以分为几类：

+ **协议层面**：客户端会先和服务端进行协议版本的协商，确保双方兼容；同时还要完成功能协商，明确到底能用哪些特性。
+ **通信层面**：消息传输采用 JSON-RPC 协议，保证数据交换的统一和可靠。
+ **工具与资源**：客户端负责发现和调用工具，也能访问和管理资源（比如文件、数据等）。
+ **提示词交互**：它还支持和提示词系统打通，帮助模型在上下文中更好地调用工具。
+ **可选功能**：包括根目录管理、采样支持等，这些不是必需的，但在一些扩展场景下会用到。
+ **运行模式**：支持同步和异步操作，灵活适配不同业务需求。
+ **传输协议**：客户端可选择不同的传输方式：
    - 基于 **Stdio** 的进程间通信（轻量快速）；
    - 基于 **Java HttpClient** 的 SSE（Server-Sent Events）传输；
    - 基于 **WebFlux** 的 SSE 客户端传输协议，用于响应式流式通信。


![](https://github.com/liuyueyi/spring-ai-demo/raw/master/docs/static/07-4.webp)

##### **MCP服务端**
在 MCP 架构中，**MCP 服务器**是最基础、也是最关键的组件。它的作用是向客户端提供工具、资源和功能，换句话说，所有对外暴露的能力基本都要通过服务端来承接。


它的主要职责可以拆解成几类：

+ **协议操作的实现**：
    - 向外暴露工具，并支持工具发现；
    - 通过 URI 提供资源管理和访问能力；
    - 提供和处理 Prompt 模板；
    - 与客户端进行功能协商，确保双方兼容；
    - 负责结构化日志记录和通知，让交互更透明。
+ **并发连接管理**：支持同时处理多个客户端请求，保证高并发下的稳定性。
+ **API 支持**：既能处理同步请求，也支持异步调用，灵活适配不同场景。
+ **传输协议实现**：
    - 基于 **Stdio** 的进程间通信（轻量、直接）；
    - 基于 **Servlet** 的 SSE 服务器传输协议；
    - **WebFlux** SSE 协议，用于响应式的流式通信；
    - **WebMVC** SSE 协议，用于传统 Servlet 的 HTTP 流式通信。


如果 MCP 客户端像是“前台的接待员”，那么 MCP 服务器就是“后端的执行中心”。它既要负责把各种工具、资源统一包装成协议定义好的接口，又要扛住并发、支持多种传输方式，保证客户端发来的请求能稳定落地。

### 二、创建一个MCP Server
那如何创建一个MCP Server，并且让大模型能够访问这个MCP Server 呢？

#### 1. 项目创建
项目创建方式与之前并无差别，创建一个SpringBoot项目，并引入SpringAI的依赖，有需要的小伙伴参考 [创建一个SpringAI-Demo工程](https://github.com/liuyueyi/spring-ai-demo/blob/master/docs/01.%E5%88%9B%E5%BB%BA%E4%B8%80%E4%B8%AASpringAI-Demo%E5%B7%A5%E7%A8%8B.md)

#### 1. 依赖配置
在pom配置文件中需要添加核心的依赖，这里是 `spring-ai-starter-mcp-server-webmvc`

```xml
<dependencies>
  <dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
  </dependency>
  <dependency>
    <groupId>org.springframework.ai</groupId>
    <artifactId>spring-ai-starter-mcp-server-webmvc</artifactId>
  </dependency>
</dependencies>
```

然后在`application.yml`配置文件中，设置MVC Server的相关参数 (下面基本都是默认配置)

```yaml
spring:
  ai:
    mcp:
      server:
        name: date-server
        version: 1.0.0
        type: SYNC
        instructions: "提供获取不同时区的当前时间，并按照北京时间进行展示"
        sse-message-endpoint: /mcp/messages
        sse-endpoint: /sse
        capabilities:
          tool: true # 是否支持工具
          resource: true # 是否支持资源
          prompt: true # 是否支持提示词
          completion: true # 是否支持补全
```

重点参数说明：

+ sse-endpoint: 用于 `Web` 传输的自定义 `SSE` 端点路径，MCP Client连接的就是这个端点
+ sse-message-endpoint: 客户端用于发送消息的端点路径，如后续大模型调用工具时，走的就是这个端点
+ type: 服务器类型（同步/异步）

#### 2. 创建一个MCP Server
对于SpringAI而言，创建MCP Server和Function call有点相似，也是通过 `@Tool` 注解将方法声明为工具，区别是一个作为mcp工具，一个是作为function call的回调方法。

```java
@Service
public class DateService {
    @Tool(description = "传入时区，返回对应时区的当前时间给用户")
    public String getTimeByZoneId(@ToolParam(description = "需要查询时间的时区") ZoneId area) {
        // 根据系统当前时间，获取指定时区的时间
        ZonedDateTime time = ZonedDateTime.now(area);

        // 格式化时间
        DateTimeFormatter formatter = DateTimeFormatter.ofPattern("yyyy-MM-dd HH:mm:ss");
        String ans = time.format(formatter);
        System.out.println("传入的时区是：" + area + "-" + ans);
        return ans;
    }
}
```

接下来，我们将这个工具注册为MCP Server 暴露给大模型。


SpringAI的自动配置将检测并注册来自以下组件的所有工具回调

+ 独立的 ToolCallback Bean
+ ToolCallback Bean 列表
+ ToolCallbackProvider Bean

工具按名称去重，每个工具名称仅保留首次出现的实例，如下创建一个`ToolCallbackProvider`的 `@Bean`

```java
@Configuration
public class ToolConfig {
    @Bean
    public ToolCallbackProvider dateProvider(DateService dateService) {
        return MethodToolCallbackProvider.builder().toolObjects(dateService).build();
    }
}
```

#### 3. 启动项目
启动项目，访问[http://localhost:8080/sse](http://localhost:8080/sse), 可以看到MCP Server已经启动了。

### 三、小结
到这里我们已经实现了一个功能齐备的 MCP Server，整体的代码非常简单，在需要作为MCP的方法上添加`@Tool`注解，然后借助`ToolCallbackProvider`将 MCP Server 注册出来即可。


本文介绍的是MCP 是通过 sse实现的，这是最简单的一种web同步请求，且没有权限控制。
