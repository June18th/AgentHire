# 14-✅14.LangGraph4j创建多Agent

沿用阿里的智能体定义，我们把基于 Spring AI ChatClient 开发的 AI 应用称为 **单智能体应用**。这类应用一般只需要一个模型 + 一组工具，逻辑相对单一，比如一个问答助手、一个写作小帮手。


但在更复杂的业务里，往往需要多个工具协同，甚至存在 **工作流切换** 的场景。举个例子：


用户输入一段评价 → 系统先判断是好评还是差评 → 如果是好评，就自动回复“感谢好评”；如果是差评，则需要进一步抽取关键信息，并把评价内容转交人工客服，跟进后续的售后维护。这类需要**条件判断 + 工具编排 + 流程切换**的应用，我们称之为 **多智能体应用**。


在 Python 生态下，很多同学会用 **LangGraph** 来构建这种复杂的智能体工作流。那在 Java 世界里，目前也有两个不错的替代方案，可以满足类似的需求。

+ [LangGraphJ](https://github.com/langgraph4j/langgraph4j)
+ [阿里智能体](https://java2ai.com/docs/1.0.0.2/get-started/agent/)

接下来我们来体验一下如何使用`LangGraph4J`来进行多智能体的开发。

#### 4. 创建工具
我们这里创建一个工具类，内部提供两个工具，分别获取当前的时间和天气。

```java
/**
 * 根据时区，返回时间的工具
 *
 * @author YiHui
 * @date 2025/8/8
 */
public class TimeWeatherTools {
    @Tool(description = "传入时区，返回对应时区的当前时间给用户")
    public String getTimeByZoneId(@ToolParam(description = "需要查询时间的时区，如Asia/Shanghai, Europe/Paris") ZoneId area) {
        // 根据系统当前时间，获取指定时区的时间
        ZonedDateTime time = ZonedDateTime.now(area);

        // 格式化时间
        DateTimeFormatter formatter = DateTimeFormatter.ofPattern("yyyy-MM-dd HH:mm:ss");
        String ans = time.format(formatter);
        System.out.println("传入的时区是：" + area + "-" + ans);
        return ans;
    }

    @Tool(description = "传入地点，返回对应地点的当前天气给用户")
    public String getWeatherByZoneId(@ToolParam(description = "需要查询天气的地区，如北京、上海") String area) {
        List<String> weathers = List.of("晴", "阴", "雨", "雪", "雷", "雾");
        String ans = weathers.get((int) (Math.random() * weathers.size()));
        System.out.println("传入的地点是：" + area + "-" + ans);
        return ans;
    }
}
```

#### 5. 创建智能体
我们直接沿用官方的demo，来创建一个`langGraph4j`实现的基于智普大模型的智能体。


![](https://github.com/liuyueyi/spring-ai-demo/raw/master/docs/static/14-1.webp)

```java
@RestController
public class ChatController {
    private final CompiledGraph<AgentExecutor.State> workflow;

    public ChatController(ChatModel chatModel) throws GraphStateException {
        workflow = AgentExecutor.builder()
        .chatModel(chatModel)
        .toolsFromObject(new TimeWeatherTools())
        .build()
        .compile();
    }
}
```

#### 6. 测试验证
在上面的Controller中，定义一个访问端点，实现智能体的访问。

```java
@RestController
public class ChatController {
    /**
     * 通过agent方式访问大模型
     *
     * @param msg
     * @return
     */
    @GetMapping("/chat")
    public Object chat(String msg) {
        AgentExecutor.State last = null;
        int i = 0;
        for (NodeOutput<AgentExecutor.State> item : workflow.stream(Map.of("messages", new UserMessage(msg)))) {
            System.out.println(item);
            last = item.state();
            System.out.printf("%02d : %s%n", i++, toStr(last.messages()));
        }

        // 返回最后一条消息
        return last.lastMessage().map(Content::getText).orElse("NoData");
    }

    public String toStr(Object obj) {
        ObjectMapper objectMapper = new ObjectMapper();
        try {
            return objectMapper.writeValueAsString(obj);
        } catch (JsonProcessingException e) {
            throw new RuntimeException(e);
        }
    }
}
```

Spring AI 的 ChatClient 是基于 **对话接口封装的**：你发一个 Prompt，它帮你走一遍调用链（包括 Advisor、工具调用、返回结果）。逻辑是单次对话 → 工具调用 → 返回。


而 **LangGraph4j** 则是一个 **对话 + 流程引擎**：它不仅能跑一次调用，还能把 **多步调用** 组织起来，像画流程图一样，把条件判断、循环、分支、节点状态都表达清楚。


换句话说，ChatClient 是“请你帮我干一次活”；LangGraph4j 是“定义一条工作流，你按图纸跑”。


工作流的核心体现在：

+ **节点（Node）**：每一步是一个节点，比如“调用工具获取时间”、“调用工具获取天气”。
+ **边（Edge）**：节点之间通过边连接，可以表达先后顺序和依赖关系。
+ **条件分支**：某个节点的输出可以决定走哪条边，例如“如果是好评 → 自动回复”，“如果是差评 → 转人工客服”。
+ **状态管理**：在长对话或多步骤任务里，可以维护上下文，不必每次都从零开始。


在 LangGraph4j 里，条件判断就是 **工作流分支**。


举个例子，刚才的场景：

1. 输入评价内容（节点 A）
2. 节点 B：调用大模型 → 判断好评/差评
3. 如果好评 → 节点 C：回复“感谢好评”
4. 如果差评 → 节点 D：抽取关键信息 → 节点 E：转人工客服


这就是典型的“条件 → 分支 → 不同执行路径”的工作流。


在 LangGraph4j 里，**Agent** 就是一个能够在工作流中 **选择工具、执行步骤** 的智能体。


+ 可以把工具注册成节点；
+ 把 LLM 的决策逻辑放在另一个节点；
+ 用边把节点串起来；
+ 最后交给 Graph 去调度执行。


所以它和 ChatClient 的最大不同在于：**不仅能对话，还能设计一条复杂的任务流**，自动跑多个步骤。

### 二、总结
**LangGraph4j** 是一个专为 Java 开发者准备的库，用来构建基于大型语言模型（LLM）的 **有状态多智能体应用**。它的灵感来源于 Python 社区的 **LangGraph**，目标是让 Java 生态下的 LLM 应用开发也能像拼积木一样顺畅。


LangGraph4j 的定位并不是替代，而是补充和扩展。它能和 **LangChain4j、Spring AI** 等主流框架无缝配合，把代理（Agent）、工具（Tool）、自定义逻辑这些组件组织成一个 **循环图（cyclic graph）**。在这个图里，不同节点可以带着状态相互通信，支持多轮、多路径的智能体协作。


本篇文章其实只是初探，演示了如何用 LangGraph4j 起手搭建一个简单的智能体项目。后面我们会逐渐深入，看看这个框架背后的设计理念，以及它在 **工作流编排、条件分支、多代理协作** 等场景中的真正威力。
