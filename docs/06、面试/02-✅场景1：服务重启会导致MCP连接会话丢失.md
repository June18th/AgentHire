## 1.问题描述
在使用校招派的MCP Server功能时，我发现大模型在调用McpServer时，有时可以正常获取推荐的内容，有时又无法获取返回结果，如下图：

![](https://cdn.nlark.com/yuque/0/2025/png/35158118/1760164215639-d77cecb9-81a0-4f40-b290-fc61dc665d80.png)



从现象来看，这个 MCP Server 的整体链路是能跑通的——因为它**有时候是能正常工作的**，说明从架构层面上并不存在致命问题，至少在部分场景下逻辑是成立的。  


但问题也很明显：**有时候又不行。**这意味着我们实现的某些环节里，存在“非确定性”的影响因子，导致执行结果出现了概率性的不稳定。



一般遇到这种“时好时坏”的问题，后端开发的第一反应往往是**线程安全或上下文污染**。多线程使用不当、请求之间的上下文混淆，是典型元凶。



不过这次情况特殊——我们的 MCP Server 实现里并没有涉及多线程处理，也不存在共享可变状态的问题。



也就是说，问题更可能出现在**调用链外部的非线程逻辑层面**：可能是请求的生命周期管理不一致、模型回调未严格遵循协议规范、或者工具注册时上下文没有正确传递。



这些不确定性问题往往比纯线程问题更棘手，因为它们不是逻辑 bug，而是“**偶发的时序干扰**”。



后续我们就得从请求调用、工具绑定、上下文注入这些节点入手，看看哪一环节在某些条件下被意外跳过或复用。

# 二、问题定位
OK，问题复现了，接下来我们就需要定位排查，为什么会出现这个问题；下面会将整个排查思路全盘记录下来。

## step1. 稳定复现
要定位问题，第一步当然是**先在本地把问题复现出来**。我们的目标很明确——让 Trae 调用校招派的 MCP Server 出现“无响应”的情况。  


幸运的是，这个问题的复现难度不高。只要保持 Trae 的 MCP Client 不变，然后**重启一次校招派服务**，问题几乎可以**稳定复现**。



也就是说，只要服务重启，Trae 就会出现连接不上 MCP Server 的现象。这一点非常关键，说明问题很可能出现在 **Server 端的初始化或注册流程** 上，而不是客户端调用逻辑。



从现象上看，Trae 的请求确实发出去了，但 MCP Server 没有做出正确响应，整个调用过程就像是“信号发出，却没人接”。  


接下来就得顺着这个方向查——看看是不是服务重启后，某些组件（比如工具注册、回调通道、或事件监听器）没有重新初始化，导致 MCP 通信链路断裂。

从后端的请求日志来看，Trae 的 MCP Client 确实成功发起了调用。服务端这边能看到的请求流程是：先收到用户的 tools/list 请求，用来获取可用工具列表；但奇怪的是，在这之后立刻就收到了 **取消通知**。  


从表现上看，这说明 **MCP Server 并没有把工具列表正确返回给 Client**。由于客户端一直没等到返回结果，它认为请求超时或无效，于是中止了后续的执行逻辑。



换句话说，整个问题的根源不在于“工具不存在”，而在于**工具列表的响应过程出错或未触发**。这也解释了为什么客户端后续所有基于工具的调用逻辑都没走下去——因为它压根没拿到初始的工具定义。  


这一步的重点排查方向，应该放在：

+ tools/list 的响应构造逻辑是否被正确执行；
+ 是否存在某种 **延迟初始化或异步加载** 的情况，导致在 Server 启动后的首次请求时，工具列表尚未注册完成；
+ MCP 协议层是否有版本或 JSON-RPC 序列化问题，使得返回的响应被客户端判定为无效。



一旦确认返回链路中断的位置，就能定位是哪一层“没说话”，从而解决这个“能听见、但不回应”的问题。

不过，这里出现了一个非常有意思的现象。在用 **Qoder** 访问校招派的 MCP Server 时，我发现它的表现和 **Trae** 并不一样。

那这中间到底发生了什么？



仔细对比请求日志之后，我发现一个关键区别：**在失败和成功之间，MCP Client 多发了一次 initialize 请求。**



这就很耐人寻味了。也就是说，Qoder 在发现调用失败后，会主动重新初始化连接，而 Trae 没有。这一步的差异，可能正是导致两者表现不同的关键。



换句话说，MCP Server 并不是彻底“挂了”，而是**在启动后首次连接时缺少初始化握手**。Qoder 的自恢复逻辑（重新发送 initialize）刚好修复了这个缺口，而 Trae 一直沿用旧连接，自然就连不上。

## step3: 确认问题根因

## step4: 复盘问题原因


 OK， 整个问题的原因算是捋顺了，我们以时序图的方式，展开看看McpClient 与 校招派McpServer的交互过程。



<!-- 这是一张图片，ocr 内容为： -->
![](https://cdn.nlark.com/yuque/0/2025/png/35158118/1760428037663-bfbaa92a-b64f-43cc-92c0-675ffa53dc35.png)



一个正常的McpServer调用流程，是先完成上面的 1-6 步骤进行初始化； 在完成这些工作之后，McpClient 和校招派的后续交互就是步骤7、8的不断重复。



一句话总结问题：



**Spring AI 实现的 MCP Server 与 MCP Client 的上下文信息是保存在 JVM 内存中的（通过 ConcurrentHashMap 维护）。**当后端服务重启后，这些会话数据会被清空，导致 MCP Server 无法识别来自旧 Client 的请求，从而无法正常响应。



更专业一点来说，问题的根源在于——**校招派后端重启后，MCP Server 与 MCP Client 的会话初始化阶段（步骤 3–6）没有重新建立**。这意味着双方的上下文状态丢失，Session 未同步恢复，最终造成客户端后续调用全部失效。  


通俗点说就是：MCP Client 还以为自己连着老朋友，实际上服务端已经“失忆”，不认识它了。

## 2.会话持久化

### 2.2 会话持久服务
接下来，我们需要实现 DAO 的上一层，<font style="color:#000000;background-color:#c7edcc;">McpSessionService</font> , 在这里，设计两个方法。一个是保存上下文。

```java
/**
     * 保存会话上下文信息
     * - 在McpClient 发送 initialize 和 notifications/initialized 方法时触发
     *
     * @param sessionId  会话id
     * @param initInfo   initialize 方法的参数
     * @param notifyInfo notifications/initialized 方法的参数
     */
    public void saveSessionInfo(String sessionId, String initInfo, String notifyInfo) {
        McpClientEntity entity = mcpClientRepository.findBySessionId(sessionId);
        if (entity == null) {
            entity = new McpClientEntity();
            entity.setSessionId(sessionId);
            entity.setUserId(ReqInfoContext.getReqInfo().getUserId());
            entity.setCreateTime(new Date());
            entity.setNotifyInfo("");
            entity.setInitInfo("");
        }
        if (StringUtils.isNotBlank(initInfo)) {
            entity.setInitInfo(initInfo);
        }
        if (StringUtils.isNotBlank(notifyInfo)) {
            entity.setNotifyInfo(notifyInfo);
        }
        mcpClientRepository.save(entity);
        log.info("保存会话上下文信息: {}", entity);
    }
```

一个是识别当前的McpServerSession，判断是否需要从持久层恢复上下文信息。

:::color2
+ **方法目的**：当 `McpServerSession` 尚未初始化时，尝试从数据库加载上下文信息并手动完成初始化。
+ **检查会话状态**：使用反射访问 `McpServerSession` 的私有字段 <font style="color:#DF2A3F;">state</font>。如果 <font style="color:#DF2A3F;">state </font>值为 0，表示会话尚未初始化。
+ **查找会话数据**：首先根据 <font style="color:#DF2A3F;">sessionId </font>查询 <font style="color:#DF2A3F;">McpClientEntity</font> 实体。若未找到匹配记录，则依据当前用户的 ID 查找最近一次会话记录作为备选方案。
+ **执行会话初始化**：如果存在有效的 <font style="color:#DF2A3F;">McpClientEntity </font>记录，解析 <font style="color:#DF2A3F;">initInfo </font>字段内容，并调用 `session.handle()` 执行初始化请求。解析 <font style="color:#DF2A3F;">notifyInfo </font>字段内容，并发送通知消息。上述操作均使用 `.block()` 等待异步处理完成。
+ **日志记录与返回结果**：成功加载上下文后记录成功日志并返回 `true`。若无可用实体则记录失败日志并最终返回 `false`。

:::



```java
/**
 * 若 McpServerSession 没有初始化，则尝试从数据库中获取上下文信息，进行手动初始化
 *
 * @param sessionId 会话id
 * @param session   会话对象
 * @return true 表示初始化成功 false 表示初始化失败
 */
public boolean autoInitSession(String sessionId, McpServerSession session) {
    try {
        // 根据 io.modelcontextprotocol.spec.McpServerSession.state 状态来判断会话是否完成了初始化
        Field field = session.getClass().getDeclaredField("state");
        field.setAccessible(true);
        AtomicInteger value = (AtomicInteger) field.get(session);
        if (value.get() == 0) {
            // 表示没有初始化，需要手动初始化，优先根据会话ID进行查询 --> 这种适用于后端分布式部署的场景
            McpClientEntity entity = mcpClientRepository.findBySessionId(sessionId);
            if (entity == null) {
                // 对于正好与客户端通过 /sse 端点建立连接的服务重启之后，因为请求断开；此时McpClient会重新请求/sse，此时会话id会发生变更
                // 为了兼容上面这种场景，这里给出一个临时解决方案，根据用户来获取最近的一条会话信息，以此来进行初始化
                entity = mcpClientRepository.findFirstByUserIdOrderByIdDesc(ReqInfoContext.getReqInfo().getUserId());
            }

            if (entity != null) {
                // 初始化
                Mono m = session.handle(JsonUtil.toObj(entity.getInitInfo(), McpSchema.JSONRPCRequest.class));
                m.block();

                // 通知
                m = session.handle(JsonUtil.toObj(entity.getNotifyInfo(), McpSchema.JSONRPCNotification.class));
                m.block();

                log.info("自动加载上下文信息成功: {} - {}", sessionId, entity);
                return true;
            } else {
                log.info("自动加载上下文信息失败: {} - {}", sessionId, ReqInfoContext.getReqInfo().getUser());
            }
            return false;
        }
        return true;
    } catch (Exception e) {
        throw new RuntimeException(e);
    }
}
```

# 四、小结
这篇内容讲的是校招派在运行中遇到的一个比较典型的问题。整个分析过程其实挺有意思的，也很有代表性。我们先从问题现象开始说起。



后端服务在重启之后，原本一直正常的 MCP Server 调用链路突然无法响应了。前端的 Trae 客户端发出的请求没有回调结果，看起来就像整个链路被掐断了一样。  


这种偶发性问题最麻烦的地方在于，它不是必现的。我们为了搞清楚根因，先尝试在本地稳定复现，结果发现只要后端服务重启，Trae 这边的 MCP 调用就一定会卡死。到这一步，基本可以确定问题和“服务重启”本身有关。  


接下来，我们考虑两种可能性：



+ 一种是 MCP Client 自身实现或使用方式有问题；
+ 另一种是 MCP Server 在重启后存在状态丢失。



为了排除干扰，我们换成 Qoder 作为客户端重新验证。第一次调用依旧失败，但奇怪的是，多试几次之后又恢复了正常。对比调用日志后发现，在失败和成功之间，Qoder 多发了一次 initialize 请求，而 Trae 从头到尾都没有发过。  


这个差异说明了一件事：Trae 一直在用老会话在请求，而 MCP Server 在重启后早就“失忆”了。Qoder 之所以能恢复正常，是因为它重新初始化了会话，相当于主动重新握了一次手。



为了彻底确认问题来源，我们又通过 debug 深入到 MCP Server 的底层逻辑。很快就发现，Spring AI 在实现 MCP Server 与 MCP Client 的上下文维护时，是直接存在 JVM 内存里的，通过一个 ConcurrentHashMap 来保存连接会话。服务一旦重启，这些数据自然就全清空了。



换句话说，Server 重启之后，Client 还以为自己连着老朋友，实际上服务端已经完全不记得它是谁。由于双方上下文没有重新初始化，后续的所有调用都被直接丢弃。  


问题的本质，其实就是会话上下文丢失造成的响应异常。解决思路也很清晰，要么让 Server 在启动时自动恢复上下文，要么让 Client 在检测到异常时主动 re-initialize，重新打一次招呼。

<!-- 这是一张图片，ocr 内容为： -->
![](https://cdn.nlark.com/yuque/0/2025/png/35158118/1760428037663-bfbaa92a-b64f-43cc-92c0-675ffa53dc35.png?x-oss-process=image%2Fformat%2Cwebp)



好，重新梳理一下问题。**假如面试官对校招派这个项目展开提问：你在做校招派 Agent 项目中，遇到了什么难题，怎么解决的？**就可以这样回答：



Spring AI 在当前版本中，将 MCP Client 的上下文信息保存在 JVM 内存里，通过 ConcurrentHashMap 进行维护。这样一来，只要应用一重启，这些上下文信息就会全部丢失。对 MCP Client 而言，它认为自己早已建立过连接，因此不会再重新发送 initialize 请求来维护连接信息。于是，后端在接收到调用请求时，发现上下文里压根没有该客户端的会话记录，自然也就无法正确响应。  


从这个根因来看，问题并不仅仅出现在服务重启的场景下。在集群部署的环境中，同样会触发类似的问题。因为只有和 MCP Client 建立连接的那台机器的内存中持有会话信息，当请求被负载均衡路由到其他机器时，这些节点上是没有上下文的，最终也会导致调用异常。  


找到问题之后，我去查了一圈，看有没有现成的解决方案。结果挺遗憾，网上几乎没人提过类似问题。于是我决定自己动手修复。我的第一反应是看官方新版本是否已经修复。查阅 Spring AI 的最新文档后发现，在 1.1.0 版本中，MCP 模块新增了 **Stateless Streamable HTTP MCP Servers**，也就是无状态的 MCP Server 协议。因为这种模式下后端不再保存会话上下文，所以自然不会再有上下文丢失的问题。



我在本地做了验证，确实可行。不过考虑到 Spring AI 1.1.0 仍处于预览阶段，未来调整空间很大，而且无状态的实现对我们这种希望保留上下文、做会话增强的业务场景并不友好，所以我暂时没有采用官方方案。



接下来我设计了自己的修复方案，思路很简单：**让会话持久化**。也就是说，把原本存在 JVM 里的会话数据，改为落地存储。实现步骤大致如下：

+ 首先，新增一个会话持久层，用数据库或 Redis 来存储上下文信息；
+ 其次，自定义一个 TransportProvider，替换掉默认实现。在 MCP Client 初始化时，将上下文信息写入持久层；
+ 最后，在 MCP Client 调用 MCP Server 时，先判断会话是否已经初始化，如果没有，就从持久层加载上下文并自动完成初始化，然后再继续执行正常的调用逻辑。



通过这几个步骤，我完整地实现了问题的复现、定位、根因分析和最终修复。整个过程下来最大的感受是——**Spring AI 在 MCP 会话这块的设计实在太简陋了**。像这种基础的状态管理功能，官方至少该预留一个扩展点，让开发者能自定义上下文的存储方式，而不是只能依赖 JVM 内存。
