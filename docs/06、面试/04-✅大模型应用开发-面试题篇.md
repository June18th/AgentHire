# 04-✅大模型应用开发-面试题篇

虽然求职派的具体实现和传统的应用开发没有明显的区别，但它的核心设计理念以及关键数据的获取都是围绕 LLM 来进行的，因此称它为LLM应用也无可厚非。


在求职派的工程中，我们提供了一个名为“求职派的 Agent”，它更类似于 AI Agent 的开发范式。为了更好的跟随时代潮流，我们很有必要学习和了解一下，AI Agent 的核心理念、主流协议（MCP、A2A）、思考框架（CoT, ReAct, Paln-And-Execute）、开发框架（LangChain、LangGraph、SpringAI）

## 一、AI Agent发展趋势
### Level-1：LLM Agent
这个属于大家接触得较多的场景，比如大模型提供的角色对话、文章润色、模拟面试等。


这个阶段的特点是，通过 **“提示词” **给智能体赋予灵魂（人设），结合LLM的多模态、OCR、ASR（语音识别）\TTS(文字转语音）能力，为用户提供服务，通常是以聊天机器人的形式存在。


但由于大模型存在幻觉，返回的信息不总是真实的，而且智能体的输出存在随机性、不可控的问题，直接通过LLM+提示词工程创建的Agent，无法很好承载严肃场景的需求。


![](https://cdn.nlark.com/yuque/0/2025/png/35158118/1756711308559-b563d800-ac96-4f71-8cd0-4410efe62c44.png)

### Level-2：AI Agent
通过引入规划、记忆、Function Call + MCP 这三个功能，实现了AI Agent对于更复杂任务的处理能力。也就是常说的：`Agent = LLM+记忆+规划+工具使用`


+ 规划：智能体能够根据给定目标，自主拆解任务步骤、执行执行计划
    - 如Dify、Coze等平台
+ 记忆：智能体具有长期记忆的能力，能够存储和调用历史信息，从而在任务执行过程中保持上下文连贯性
+ 工具使用：LLM虽然在信息处理方面表现出色，但它们缺乏直接感知和影响现实世界的能力，工具是LLM连接外部世界的关键，智能体能够通过使用工具，例如调用API、访问数据库等等，与外部世界进行交互
    - 如 Function Call, MCP协议都是用于增强大模型能力边界的方式

Agent与大模型对比

简单翻译一下：

| | **模型（Models）** | **智能体（Agents）** |
| --- | --- | --- |
| 知识范围 | 知识局限于训练数据中已有的内容。 | 通过工具与外部系统连接，从而扩展知识。 |
| 推理能力 | 基于用户查询进行单次推理/预测。除非明确为模型实现相关功能，否则不会管理会话历史或连续上下文（如聊天历史）。 | 管理会话历史（如聊天历史），以支持基于用户查询和编排层中所作决策进行多轮推理/预测。在此上下文中，“一轮”被定义为交互系统与智能体之间的一次交互（即 1 个传入事件/查询和 1 个智能体响应）。 |
| 工具扩展 | 没有原生工具实现。 | 工具在智能体架构中是原生实现的。 |
| 逻辑处理 | 没有原生逻辑层实现。用户可以将提示词编写为简单问题，或者使用推理框架（如 CoT、ReAct 等）来编写复杂提示词，以引导模型进行预测。 | 具有原生认知架构，该架构使用诸如 CoT、ReAct 之类的推理框架，或像 LangChain 这样其他预先构建的智能体框架。 |


### Level-3：Multi Agent
单个agent适用于较单一的场景，为了不再依赖单一的"全能型"Agent，接下来就是朝着构建多个专业化Agent组成的协作网络的`Multi-Agent`发展了 （看起来很像单体 -> 微服务的演进路线）

> 如Meta开源的 MetaGPT [https://github.com/FoundationAgents/MetaGPT/blob/main/docs/README_CN.md](https://github.com/FoundationAgents/MetaGPT/blob/main/docs/README_CN.md)
>


![](https://cdn.nlark.com/yuque/0/2025/png/35158118/1756712865280-e91fa56d-de13-4244-80e0-708508950e09.png)


Mult-Agent的核心优势：

+ **任务聚焦**：单个智能体专注特定任务（如搜索或图表生成），比多工具选择更高效。
+ **独立优化**：可单独改进单个智能体，不影响整体流程。
+ **问题拆解**：将复杂问题拆分为可处理的子任务，由专业智能体处理。


此外，人的参与互动，也可以作为一个特殊类型的Agent，加入到智能体的协作当中。


在多Agent系统中，人的参与具有独特且不可替代的价值:

1. **专业判断**: 在关键决策节点，人可以基于丰富的经验和领域知识，提供更可靠的判断和指导。
2. **质量把控**: 人可以审核和验证AI Agent的输出结果，确保其符合业务需求和质量标准。
3. **异常处理**: 当AI Agent遇到无法处理的复杂情况时，人可以介入并提供解决方案。
4. **持续优化**: 通过观察AI Agent的表现，人可以识别系统的不足，并对Agent的能力边界和工作流程进行调整和优化。

## 二、Agent协议
简单来说，LLM应用开发有下面三种：

+ 用户与大模型交互：LLM 标准接口（可以理解为大模型的请求、返回，由于OpenAI的先驱特性，现在大多数的大模型都是兼容OpenAI的接口交互）
+ 大模型与工具之间的使用：Function Call / MCP协议
+ Agent与Agent之间： A2A协议


![](https://cdn.nlark.com/yuque/0/2025/png/35158118/1756714149623-bdf4c325-9cbb-42a3-babf-7a9b0961cb5e.png)

### 1 LLM标准接口
这个可以理解为大模型应用开发，对接 LLM 厂商提供的标准开放接口。


LLM标准接口，主要解决的是用户如何与大模型进行交互/信息传输，通常有同步、异步两种，都是基于 HTTP 协议的。


**同步：**向LLM发送请求，等待LLM全部一次性返回结果

**异步：**

1. 基于SSE的方式，LLM分批将结果返回给客户端，这样给人的感觉就像打字机一样，不断的输出结果
2. 返回任务id，后台轮询查询任务执行结果（常见于图片、音视频生成等耗时的场景，LLM需要较长的处理时间）
    1. 用户发起请求
    2. LLM返回一个任务id
    3. 用户周期性的拿这个任务id，到LLM这里查询任务结果
    4. LLM根据实际情况，返回 运行中、运行失败、运行成功等状态


### 2 大模型上下文协议：MCP
> [https://modelcontextprotocol.io/docs/getting-started/intro](https://modelcontextprotocol.io/docs/getting-started/intro)
>

全称为（Model Context Protocol）：由Anthropic 于24年11月底推出，号称AI届的USB，主要是为了解决 LLM 如何从外部世界（数据、工具、服务）获取完成任务所需信息（上下文）的问题


![](https://cdn.nlark.com/yuque/0/2025/png/35158118/1756715707061-f8eaf7cf-dd54-48b3-8fad-5859c300d60d.png)

_图片来自：_[_https://zhuanlan.zhihu.com/p/1919338285160965135_](https://zhuanlan.zhihu.com/p/1919338285160965135)

在MCP架构中，有三个重要的角色

+ MCP Hosts(MCP主机)： 通过mcp协议获取数据的程序（比如 Claude Desktop, IDE 或者 ai工具）
+ MCP Clients(MCP客户端): mcp协议客户端，作为调用者，通常与mcp服务端成对出现，用于连接mcp server，实现双方通信
+ MCP Servers(MCP服务端): mcp协议服务端，作为提供者，实现MCP协议提供特定的功能

除了上面三个角色之外，在架构图中，还有外部资源

+ Local Data Sources: MCP Server所在的计算机上本地的资源信息，如文件、数据库、应用服务等，即MCP通过操作(读取、访问、写入等)这些本地资源来提供特定能力
+ Remote Service: MCP Server通过调用远程的服务来提供能力


基于MCP协议，Agent可以不局限于语言、框架，集成社区里优秀的MCP Server，让Agent方便调用外部API、访问和修改各类资源，实现如自动化办公、数据抓取、信息检索、流程编排、跨系统集成等多种能力。


MCP协议的标准化设计，使得工具的开发与接入变得高度便捷，极大提升了Agent的可扩展性和生态兼容性。开发者可以快速复用社区已有的工具组件，降低开发和维护成本，同时也让Agent具备了灵活适配不同业务场景的能力，推动了AI Agent在实际生产环境中的落地和普及。

### 3 Agent间协议：A2A
> [https://a2a-protocol.org/latest/topics/what-is-a2a/](https://a2a-protocol.org/latest/topics/what-is-a2a/)
>

全称Agent2Agent， A2A是由Google于25年4月发起的一个开放协议，旨在促进不同AI 代理(Agent) 之间的互通，其工作原理：


![](https://cdn.nlark.com/yuque/0/2025/png/35158118/1756716314932-8a29abe1-1acf-42c8-88f8-196d4878ab08.png)

+ **能力发现**：智能体通过JSON格式的`Agent Card`宣传自身能力，便于其他Agent发现和调用最合适的智能体。
+ **任务和状态管理**：以任务为导向，协议定义了任务对象及其生命周期，支持任务的分发、进度同步和结果（工件）输出，适配长短任务场景。
+ **协作与消息交换**：智能体之间可发送消息，交流上下文、回复、工件或用户指令，实现多智能体间的高效协作。
+ **用户体验协商**：每条消息可包含不同类型的内容片段（如图片、表单、视频等），支持客户端和远程Agent协商所需的内容格式和UI能力。


基于A2A协议，开发者可以构建与其他A2A智能体互联互通的系统，实现跨平台、跨厂商的多Agent协作。


例如在企业招聘场景中，一个Agent负责筛选候选人，另一个Agent负责安排面试，第三个Agent负责背景调查，所有流程通过A2A协议无缝衔接，大幅提升自动化和协作效率。A2A协议的开放性和标准化为未来智能体生态的互操作性和创新能力奠定了坚实基础。

### 4 协议关键对比
#### a. Function Call 与 MCP
:::danger
一句话总结：Function Call不是协议，是大模型提供的一种能力，MCP里的工具调用，是基于Function Call能力实现的，对于工具来说，MCP和Function Call是依赖关系。但MCP除了工具外，还有Prompts、Resources等其他上下文的定义。

:::


Function Call 是大模型提供的基础能力，允许模型根据自然语言指令自动调用预定义函数，实现与外部工具的连接，但它并非统一标准 —— 不同厂商（如 OpenAI、DeepSeek、智谱）对其接口格式、参数定义等有独立实现，导致工具需针对不同模型重复适配，类似各品牌手机自有充电接口的碎片化问题


MCP（模型上下文协议）则是跨模型、跨工具的统一交互标准，不仅规范了工具调用（如函数描述、参数格式），还整合了 Prompts、资源管理等上下文体系，目标是成为 AI 生态的 "USB-C"，让工具只需按统一协议封装一次，即可在多模型中通用，大幅降低跨平台适配成本。


MCP同样面临一些问题，如

+ **生态成熟度不足**：MCP 应用市场虽有超 1.3 万个工具（MCP Server），但多数存在配置复杂、实现不规范、同质化严重等问题，真正能直接用于生产环境的少之又少，开发者常因适配成本高而选择直接调用 API；
+ **企业基建冲突**：若团队已有统一的工具调用体系（如自研 Agent 框架、API 网关），MCP 的协议层可能被视为冗余，现有基建已实现工具管理、监控等功能，引入 MCP 反而增加运维负担，类似服务网格在成熟基建中难以落地的困境；
+ **通用协议的场景局限**：MCP 的标准化设计难以满足金融、工业等垂直领域的定制需求（如安全审计、数据隔离），此时直接开发专用工具链反而更高效。


总结两者的区别在于：

+ Function Call属于大模型的基础能力，用于连接外部工具、资源，用于增强模型能力的一种手段
+ MCP则是为了推动跨模型、跨生态的能力/工具复用而设计出来的共识协议，通常它是基于Function Call来实现


#### b. MCP 与 A2A
:::danger
一句话总结：MCP主要是面向Agent内，通过标准化的工具接口设计，方便Agent更好的与外部交互；而A2A则主要针对多个Agent之间，如何更好的协同工作、上下文传输。两者没有绝对的分界线、更多选型区别在于应用场景的定位上：从协议分层来说，MCP、A2A并非互斥，而是分层协同

:::

MCP主要解决"Agent如何用好工具"，通过标准化工具接口，极大提升了工具复用和生态兼容性。

而A2A则解决"多个Agent如何协作"，通过标准化Agent间通信，推动了多Agent系统的互操作和协作创新

|   | MCP | A2A |
| --- | --- | --- |
| 重心 | MCP规范了人工智能模型和代理如何连接到工具、API、数据源和其他外部资源并与之交互 | A2A 针对独立且通常不透明的 AI 代理之间的通信和协作方式设定了通义的标准 |
| 机制 | 它定义了一种结构化的方式来描述工具功能（类似于大型语言模型中的函数调用），向它们传递输入，并接收结构化输出。 | 它为代理提供了一个应用级协议，以便：  + 发现彼此的高级技能和能力（通过代理卡）。  + 协商交互模式（文本、文件、结构化数据）。  + 管理共享、有状态且可能长时间运行的任务。  + 交换对话上下文、指令以及复杂的多部分结果 |
| 应用案例 | + 使LLM能够调用外部API（例如，获取当前股价）。  + 允许代理使用特定参数查询数据库。  + 将代理连接到一组预定义的功能或服务。 | + 客服代理将复杂的账单查询委托给专门的账单代理，并维护客户交互的上下文。  + 旅行计划代理与不同的航班、酒店和活动预订代理进行协调，管理多阶段预订流程。  + 代理为随时间推移而发展的协作项目交换信息和状态更新。 |
| 目标 | MCP旨在创建一个生态系统，工具提供商可以轻松地将他们的服务暴露给各种AI模型和代理框架，代理开发人员可以轻松地以标准化的方式使用这些工具。 | 与简单的工具调用相比，A2A 允许更动态、更有状态且可能支持多模式的交互。使用 A2A 的代理作为代理（或代表用户）进行通信，而不仅仅是调用离散函数。 |


A2A 和 MCP 并不相互排斥；它们高度互补，并满足Agent在不同层面的交互需求；如Agent内部使用MCP于特定的工具资源进行交互、Agent之间使用A2A进行相互通信。同样的若我们将Agent内部的某个工具调用也定义为一个Agent，这也可看成是MCP也用于Agent之间的交互了。


再举个实际的例子，假如 AI Agent 需要完成一个"规划 5 天深圳到厦门旅行"，使用 MCP 和使用 A2A 的都可以实现：


![](https://cdn.nlark.com/yuque/0/2025/png/35158118/1756718072328-def0248c-5c44-4bff-8896-b2c0dda1eaf4.png)


+ MCP：中心调度者。一个中央 Agent (MCP Travel Client) 负责调用所有外部服务（机票、酒店、天气），然后汇总信息生成计划。优点是简单可控，缺点是中心化依赖高，不易扩展。
+ A2A：协作共创。任务被分配给专门的 Agent（交通、住宿、活动），这些 Agent 可以直接相互沟通（比如机票 Agent 直接问天气 Agent 获取信息），也可以和用户进行沟通（比如机票Agent完成初筛之后询问用户是否满足需求，对用户给出的建议进行迭代修改），这种方式更灵活，适合企业内复杂协作。


那么在实际的项目开发中，如何进行抉择呢？

+ MCP会更适合那些交互模式类似于传统API调用的场景
    - Agent作为工具执行者，关注明确的输入和输出，且交互流程相对固定时，MCP的结构化和工具导向特性会更具优势
+ A2A更适用于需要多个Agent进行复杂协作、对话式交互和任务共同完成的场景
    - 系统需要Agent之间进行动态协商、分工合作，并且结果的达成比固定的交互流程更重要


## 三、Agent思考框架
构建能够自主规划、执行和适应复杂任务的智能体，其核心在于其"思考"能力。


Agent思考框架，正是为了赋予Agent这种结构化的推理和决策能力而设计的。这些框架提供了一套方法论，指导Agent如何理解目标、分解任务、利用工具、处理信息、并根据环境反馈调整行为

### 1 思维链：CoT
思维链（Chain of Thought, CoT）是一种增强大型语言模型（LLM）处理复杂推理任务能力的关键技术。其核心在于引导模型在给出最终答案前，先生成一系列结构化的中间推理步骤——这如同模拟人类解决问题时的逐步思考过程。


通过这种方式，LLM能够更深刻地理解问题结构，有效分解复杂任务，并逐步推导出解决方案。


这些显式的思考步骤不仅为模型的决策过程带来了透明度和可解释性，也方便用户理解和调试。


但是这种方法的代价是，生成冗长的思考链条会增加计算成本和处理延迟。

#### 一个简单的例子：没有CoT vs 有CoT
**问题：** 餐厅里有10张桌子，每张桌子有4条腿。请问餐厅里所有桌子总共有多少条腿？

##### 1. 没有思维链的传统方式：
AI可能会直接凭直觉反应：“10张桌子，每张4条腿... 嗯... 大概是40条吧！” 它直接跳到了答案。虽然答案是对的，但如果问题更复杂，它就很容易出错。

##### 2. 使用思维链（CoT）的方式：
AI会像小学生解应用题一样，**把推理过程“说”出来**：

“好的，我们来一步步解决这个问题：

1. 首先，我知道有 **10张** 桌子。
2. 然后，每张桌子有 **4条** 腿。
3. 所以，要计算总腿数，我应该用 **桌子的数量** 乘以 **每张桌子的腿数**。
4. 也就是：10 × 4。
5. 10 × 4 = 40。

因此，餐厅里所有桌子总共有 **40条** 腿。”

你看，这个过程就是“思维链”。它不是一个无声的、瞬间的直觉，而是一条有逻辑的、可见的“思考链条”。


### 2 推理系统：ReAct
:::danger
**一句话概括：ReAct 让AI不仅能在脑子里一步步推理（Reasoning），还能主动采取“行动”（Acting）去获取外部信息（比如上网搜索、查数据库、使用工具），从而解决更复杂、需要最新知识或具体数据的问题。**

:::

ReAct（Reasoning and Action）框架通过将"推理"（Reasoning）与"行动"（Action）相结合，解决CoT场景下，由于其推理过程局限于模型内部知识，缺乏与外部世界的实时交互，这可能导致知识陈旧、产生幻觉或错误传播的问题。


ReAct 允许模型在推理过程中与外部工具或环境进行互动，从而获取最新信息、执行具体操作，并根据反馈调整后续步骤；动态的交互赋予了模型一种"边思考边行动、边观察边调整"的能力，其核心运作机制可以概括为`思考（Thought）→ 行动（Action）→ 观察（Observation）`的迭代循环

+ **思考 (Thought)：**模型基于当前任务目标和已有的观察信息，进行逻辑推理和规划。它会分析问题，制定策略，并决定下一步需要执行什么动作（例如，调用哪个工具、查询什么信息）来达成目标或获取关键信息。
+ **行动 (Action)：**根据"思考"阶段制定的计划，模型生成并执行一个具体的行动指令。这可能包括调用外部API、执行代码片段、查询数据库，或者与用户进行交互等。
    - `**Search[关键词]**`：在网络上搜索最新信息。
    - `**Lookup[条目]**`：在知识库或文档中查找具体内容。
    - `**Calculate[算式]**`：使用计算器工具进行计算。
    - `**Execute[代码]**`：运行一段代码来获取结果。
+ **观察 (Observation)：**模型接收并处理"行动"执行后从外部环境（如工具的返回结果、API的响应、用户的回复）中获得的反馈信息。这些观察结果将作为下一轮"思考"的输入，帮助模型评估当前进展、修正错误、并迭代优化后续的行动计划，直至任务完成。


![](https://cdn.nlark.com/yuque/0/2025/png/35158118/1756720148819-a621c4b9-8b37-4aed-bc56-41b61201db61.png)

#### 一个简单的例子：没有ReAct vs 有ReAct
**问题：** “梅西目前效力的球队是哪支？这支球队最近一场比赛的比分是多少？”

##### 1. 只有思维链（CoT）的AI（知识截止于2023年初）：
+ **推理：** “我知道梅西在2023年之前效力于巴黎圣日耳曼，但现在可能转会了。我不知道他現在在哪。我也不知道比分，因为需要最新数据。我无法准确回答。”
+ **结果：** 失败。因为它的知识是旧的，且无法主动更新。

##### 2. 使用ReAct的AI：
它会生成一个类似下面的过程：

**推理1：** “要回答这个问题，我需要先知道梅西现在在哪个队。我的内部知识可能过时了，我应该先去网上搜索最新消息。”

**行动1：** `**Search[梅西当前效力球队 2025年]**`

**观察1：** [搜索引擎返回：利昂内尔·梅西目前效力于美国职业足球大联盟的迈阿密国际队。]

**推理2：** “好的，我知道了他在迈阿密国际。现在我需要找到这支球队最近一场比赛的比分。我应该搜索迈阿密国际最近的赛果。”

**行动2：** `**Search[迈阿密国际最新比赛比分]**`

**观察2：** [搜索引擎返回：迈阿密国际最近一场比赛是2025年8月31日对阵纽约城FC，比分是2-1获胜。]

**推理3：** “现在我有了所有信息。我可以组织答案了。”

**最终答案：** “梅西目前效力于迈阿密国际队。该队最近一场比赛是在2025年8月31日对阵纽约城FC，以2-1获胜。”


#### CoT vs. ReAct
| **特性** | **思维链 (CoT)** | **ReAct** |
| --- | --- | --- |
| **核心** | **内部推理** | **推理 + 外部行动** |
| **过程** | 在脑海中一步步思考 | 思考 -> 行动 -> 观察 -> 再思考... |
| **依赖** | 仅依赖模型已有的知识 | 依赖模型知识 + **外部工具**（搜索、计算器等） |
| **好比** | **闭门解题的学霸** | **会查资料、会用计算器的侦探** |
| **适用场景** | 主要解决逻辑推理、数学题等 | 需要**最新信息**或**特定工具**的复杂任务 |


### 3.规划执行：Plan-and-Execute
:::danger
一句话概括：Plan-and-Execute 的核心思想是：先花时间制定一个完整的“总体规划”，然后再严格地按计划一步步执行，而不是边想边做。

:::

Plan-and-Execute 是一种对标准 ReAct 框架的扩展和优化，旨在处理更复杂、多步骤的任务。它将 Agent 的工作流程明确划分为两个主要阶段

+ **规划阶段**：Agent 首先对接收到的复杂任务或目标进行整体分析和理解。然后，它会生成一个高层次的计划，将原始任务分解为一系列更小、更易于管理的子任务或步骤。这种分解有助于在执行阶段减少处理每个子任务所需的上下文长度，这个计划通常是一个有序的行动序列，指明了要达成最终目标需要完成哪些关键环节。这个蓝图可以先呈现给用户，允许用户在执行开始前对计划步骤给出修改意见。
+ **执行阶段**：计划制定完成后（可能已采纳用户意见），Agent 进入执行阶段。它会按照规划好的步骤逐一执行每个子任务。在执行每个子任务时，Agent 可以采用标准的 ReAct 循环来处理该子任务的具体细节，例如调用特定工具、与外部环境交互、或进行更细致的推理。执行过程中，Agent 会监控每个子任务的完成情况。如果某个子任务成功，则继续下一个；如果遇到失败或预期之外的情况，Agent 可能需要重新评估当前计划，可以动态调整计划或返回到规划阶段进行修正。此阶段同样可以引入用户参与，允许用户对子任务的执行过程或结果进行反馈，甚至提出调整建议。

#### 一个生动的例子：策划一场生日派对
**任务：** “帮我为一位10岁的小朋友策划一场恐龙主题的生日派对。”

##### 1. 标准 ReAct 方式（可能很混乱）：
AI可能会开始：

**推理1:** “首先，我需要确定派对主题... 用户已经说了是恐龙主题。好，接下来需要定一个日期和地点。”
**行动1:** `**Search[本地适合办生日派对的场地]**`
**观察1:** [返回了20个场地信息]
**推理2:** “天啊，这么多场地！我得先看预算... 用户还没说预算。我应该先问预算吗？还是先想游戏？...”

你看，它很容易陷入细节，或者因为信息不足而卡住，需要来回跳跃。

##### 2. Plan-and-Execute 方式（清晰高效）：
**阶段一：计划（由“计划者”LLM完成）**

“好的，策划一场生日派对通常包含以下关键步骤，我将为之制定一个计划：

1. **确定预算和人数**：这是所有决策的基础。
2. **研究和预订场地**：根据预算和人数，寻找恐龙主题或适合装饰的场地。
3. **规划菜单和订购蛋糕**：设计符合主题的儿童餐点和恐龙形状的蛋糕。
4. **准备活动和游戏**：设计如‘挖掘恐龙化石’、‘恐龙蛋寻找’等主题游戏。
5. **购买装饰和用品**：采购恐龙气球、桌布、邀请函等。
6. **发送邀请**”

_输出：一个清晰的6步计划列表。_

**阶段二：执行（由“执行者”LLM或工具逐条完成）**

**执行步骤1:** “现在执行第1步：确定预算和人数。” -> _（可能会生成一个问题让用户回答）_
**执行步骤2:** “现在执行第2步：研究和预订场地。已知预算为X元，人数为Y人。” -> `**Search[本市 儿童派对场地 恐龙主题 可容纳Y人 预算X元]**`
**执行步骤3:** “现在执行第3步：规划菜单...” -> `**Search[儿童派对餐点创意 恐龙主题蛋糕]**`
... （依此类推，直到所有步骤完成）

整个过程有条不紊，每一步都目标明确，不容易跑偏。

#### ReAct vs. Plan-and-Execute
| **特性** | **标准 ReAct** | **Plan-and-Execute** |
| --- | --- | --- |
| **工作模式** | **交替进行**：推理 -> 行动 -> 推理 -> 行动 ... | **线性进行**：**计划** -> 执行 -> 执行 -> 执行 ... |
| **核心思想** | **边想边做**，在行动中调整策略 | **先想后做**，谋定而后动 |
| **优点** | 灵活，能根据新信息实时调整 | **更可靠**（减少错误累积）  **更高效**（调用LLM次数更少）  **更擅长处理复杂任务** |
| **缺点** | 容易在长任务中迷失，效率较低 | 如果初始计划有重大缺陷，可能全部白干。灵活性稍差。 |
| **好比** | **边走边问路的探险家** | **先查好完整地图再出发的司机** |
